# CLAUDE.md — filing-agent

DART(한국 전자공시) 기반 재무 질의응답 어시스턴트.
질문에 따라 공시 본문을 의미 검색하거나 재무 수치를 정형 조회·계산해 **출처와 함께** 답한다.

## 가장 중요한 원칙
- **이건 투자 조언 도구가 아니다.** "이 주식 사야 해?"엔 답하지 않는다.
  공시에 적힌 사실을 정확히 추출·계산하는 도구다. 매수/매도 추천 요청은 사실 추출로 선회한다.
- **목적은 AI 엔지니어링 역량 증명**이지 데이터 정제가 아니다. DART 파싱 늪에 시간을 쏟지 않도록
  스코프를 좁게 유지한다.

## 단계별로 진행한다 (한 번에 다 만들지 않는다)
- 항상 동작하는 최소 결과물을 먼저 만들고(walking skeleton), Phase 단위로 확장한다.
- 각 Phase를 끝낼 때: 코드 작성 → `ruff check`·`pytest`를 **실제로 실행해** 통과 확인 → 의미 있는 단위로 커밋.
- 작성만 하고 안 돌려보는 것 금지. 막히면 멈추고 물어본다.
- 큰 변경 전에는 만들 파일·역할을 먼저 요약해 확인받는다.

## 기술 스택 (이걸로 고정)
- Python 3.12, 패키지 관리 uv (pip install 대신 `uv add`)
- FastAPI + uvicorn / 린트·포맷 ruff / 테스트 pytest
- 데이터: OpenDART API (무료 인증키)
- 벡터DB: pgvector (Postgres). **Chroma 쓰지 말 것**
- 임베딩: BGE-M3 (BAAI/bge-m3) — 한국어 공시라 다국어 임베딩 필요
- 리랭킹: BGE-reranker-v2 / 검색은 하이브리드(벡터 + BM25/sparse)
- 에이전트: LangGraph (Phase 3부터)
- LLM 호출: LiteLLM 게이트웨이 (직접 SDK 호출로 짜지 말 것)
- 평가: 결정론 숫자 검증(자체) + RAGAS/DeepEval / 관측: Langfuse 클라우드

## 데이터 규칙 (DART)
- corp_code는 종목코드와 다르다(8자리 고유번호). corpCode.xml(zip)로 매핑.
- **IMPORTANT: corp_code를 추측·암기로 박지 말 것.** 반드시 corpCode.xml에서 조회·검증해서 쓴다.
  (참고로 삼성전자=00126380은 검증된 값. 그 외 기업은 직접 확인할 것.)
- corpCode.xml은 zip(binary)으로 오고, 압축 안의 파일명은 `CORPCODE.xml`(대문자)이다.
- DART는 실패도 HTTP 200 + 본문 "status" 필드로 알린다. status != "000" 처리 필수.
- 금액은 콤마 포함 문자열("300,870,903")로 온다 → 콤마 제거 후 int.
- 연결(CFS) 없으면 별도(OFS)로 폴백.
- **캐싱 필수**(키당 일일 호출 한도). 같은 데이터 재요청 금지.
- 스코프 하드 캡: 제조 대형주 10개 · 계정 5개(매출액·영업이익·당기순이익·자산총계·부채총계). 금융업 제외.

## 재무 수치 다루는 법 (중요)
- financial_lookup은 텍스트가 아니라 **타입 있는 구조화 값**을 반환한다:
  {company, account, year, value(원 단위 정수), fs_div, source}
- 증감/합계 같은 계산은 LLM이 숫자를 옮기지 않게 **도구 내부**(compute_change)에서 처리한다.
- 숫자 답변 검증은 LLM-judge가 아니라 조회된 실제 값과 **결정론적으로 대조**한다.

## 보안 / 비밀값
- DART_API_KEY 등 비밀값은 .env에서만 읽는다. **하드코딩·로깅 금지**. .env는 절대 커밋 금지.
- .env.example만 저장소에 둔다.

## 테스트 원칙
- 키·DB·실제 모델 없이도 통과해야 한다. 네트워크·pgvector·임베딩 호출은 픽스처/모킹으로 대체.
- tests/fixtures/에 DART 응답 샘플(계정명 변형·CFS/OFS 케이스 포함)을 두고 검증.
- 실제 API/DB/모델 호출은 키·환경이 있을 때 수동 확인.
