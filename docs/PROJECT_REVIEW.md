# 🔍 filing-agent 프로젝트 기술 검토 및 개선 제안 보고서

본 보고서는 금융감독원 전자공시시스템(OpenDART) 데이터를 기반으로 재무 수치 및 공시 서술 내용을 분석하여 신뢰성 있는 답변을 제공하는 **filing-agent** 프로젝트의 현재 구현 상태(Phase 0)를 분석하고, 기술 스택 검토, 개선사항, 그리고 향후 추가하면 좋은 기능적/아키텍처적 요소를 정리한 검토서입니다.

---

## 1. 프로젝트 개요 및 현재 상태

### 1.1 프로젝트 목표
*   **핵심 가치**: LLM의 환각(Hallucination) 현상을 방지하기 위해 공시 데이터를 근거로 **출처가 명시된 사실(Facts)만**을 사용자에게 제공합니다. (투자 추천 배제, 사실 추출에 집중)
*   **아키텍처 모델**: 정형 재무 데이터(주요 계정)와 비정형 텍스트(사업보고서 본문)를 분리하여 처리하는 하이브리드 라우팅 에이전트(LangGraph 기반)와 검증 루프(Harness) 체계를 지향합니다.

### 1.2 현재 구현 상태 (Phase 0)
*   프로젝트의 기초 뼈대(FastAPI, 린터/포맷터 Ruff, 테스트 Pytest)가 구축되어 있습니다.
*   단일 기업(삼성전자) 및 단일 계정(매출액)에 대해 OpenDART API를 직접 조회하여 결과를 반환하는 결정론적(Deterministic) `/ask` API가 구현되어 있습니다.
*   가상환경 및 의존성 관리에 Astral사의 최신 도구인 `uv`를 사용하고 있습니다.

## 3. 현재 구현 코드의 개선사항 (Current Code Improvements)

현재 Phase 0 수준의 코드에서 즉각 개선할 수 있는 기술적 부채와 최적화 포인트입니다.

### 3.1 동기식 HTTP 요청을 비동기식(Async)으로 전환
*   **문제점**: `src/filing_agent/ingest/dart_client.py`는 `httpx.get`을 동기적으로 호출하고 있습니다. FastAPI는 비동기(Async) 프레임워크인데, 동기식 I/O 작업을 수행하면 대량의 동시 요청이 들어왔을 때 이벤트 루프가 차단(Block)되어 서버 처리량이 급격히 떨어집니다.
*   **개선 방안**: `httpx.AsyncClient`를 도입하고, `async def` 기반으로 클라이언트를 재작성해야 합니다.

### 3.2 connection pool 재사용 및 FastAPI Lifespan 활용
*   **문제점**: API 호출(`resolve_corp_code`, `fetch_single_account`)마다 매번 신규 HTTP Connection을 생성하고 닫고 있어 네트워크 핸드셰이크 비용이 낭비됩니다.
*   **개선 방안**: FastAPI의 `lifespan` 이벤트를 사용하여 앱 시작 시 `httpx.AsyncClient` 세션을 한 번만 생성하여 전역적으로 재사용(Connection Pooling)해야 합니다.

### 3.3 동기식 파일 기반 I/O 캐싱 개선
*   **문제점**: JSON 캐시를 쓰고 읽는 작업(`Path.write_text`, `Path.read_text`)이 모두 동기식 블로킹 코드로 짜여 있습니다.
*   **개선 방안**: 로컬 캐싱 시 `anyio` 또는 `aiofiles`를 활용하여 비동기 파일 I/O를 수행하거나, 보다 강력한 캐시 메커니즘을 위해 SQLite(단일 로컬 파일 DB)나 Redis(메모리 캐시) 도입을 검토해야 합니다.

### 3.4 데이터 유효성 검사 및 보안 가드레일 추가
*   **문제점**: 현재 `/ask` 라우트에서 `company` 쿼리 매개변수를 아무런 검증 없이 그대로 DART API 요청에 전달합니다. 악성 입력이나 비정상적인 파라미터가 유입될 경우 보안 취약점이 될 수 있습니다.
*   **개선 방안**: Pydantic을 활용해 입력 파라미터(`company` 이름의 길이 및 특수문자 제한, `year`의 유효 범위 제한 등)를 엄격히 검증하는 가드레일을 주입해야 합니다.

## 4. 향후 추가하면 좋은 추천 기능 및 아키텍처 확장 (Recommended Future Features)

프로젝트의 실무적 가치를 극대화하고 이력서/포트폴리오 완성도를 차별화하기 위해 향후 추가해야 할 핵심 기능들입니다.

### 4.1 IFRS 표준 계정과목 매핑 레이어 구축 (Taxonomy Alignment)
*   **필요성**: DART 공시 자료에서 기업마다 계정과목명을 서로 다르게 기재하는 경우가 많습니다. (예: 어떤 기업은 매출액을 "매출액", 어떤 기업은 "영업수익", 금융/서비스는 "수수료수익" 등으로 기재)
*   **구현 방안**: 
    1. 금융감독원의 K-IFRS 표준계정과목(Taxonomy) 코드를 파싱하여 대표 계정과목명으로 매핑하는 정규화 규칙 사전 구축.
    2. 동의어 매핑 테이블을 구현하고, 예외적인 표현은 LLM을 통해 표준 계정명(매출액, 영업이익 등)으로 시맨틱 맵핑(Semantic Mapping)해주는 도구 레이어 추가.

### 4.2 대화 컨텍스트 유지 기능 (Multi-turn Conversational Memory)
*   **필요성**: 현재는 "단일 질문-단일 답변" 구조입니다. 사용자가 "삼성전자 2024년 매출액 알려줘"라고 물어본 뒤, "그럼 영업이익은 얼마야?" 또는 "작년 대비 얼마나 늘어난 거야?"와 같이 앞선 맥락을 이어 질문하는 경우가 많습니다.
*   **구현 방안**: LangGraph 내에 `MemorySaver` 또는 Postgres 기반 `SqliteSaver` / `PostgresSaver`를 연동하여 세션 ID별 대화 상태(State)와 이전 조회 결과(`FinancialFact`)를 영속화하여 컨텍스트 기반 답변을 하도록 확장합니다.

### 4.3 에이전트 추론 과정 스트리밍 및 SSE(Server-Sent Events) 지원
*   **필요성**: LangGraph 에이전트가 복잡한 하이브리드 질문을 처리할 때(검색 -> 도구 호출 -> 계산 -> 검증), 최종 답변이 나오기까지 5~10초 이상의 지연이 발생할 수 있습니다. 사용자 경험(UX) 관점에서 진행 상황이 보이지 않으면 사용자는 중단된 것으로 느낄 수 있습니다.
*   **구현 방안**: FastAPI에서 `StreamingResponse`를 활용하여 에이전트의 노드 실행 상태(예: "삼성전자 공시 본문 검색 중...", "재무제표 수치 계산 중...") 및 LLM 응답 토큰을 실시간으로 화면에 출력할 수 있도록 이벤트 스트리밍 규격을 설계합니다.

### 4.4 대규모 API 호출 한도 대응을 위한 로컬 캐시 DB 구축
*   **필요성**: OpenDART API는 하루 10,000회라는 엄격한 호출 제한이 있습니다. 사용자가 몰리거나 대규모 평가(Evaluation)를 위해 100개 이상의 테스트 케이스를 동시에 돌릴 때 API 제한에 걸릴 위험이 큽니다.
*   **구현 방안**: 
    *   다운로드받은 주요 재무 수치와 `corpCode` XML 데이터를 디스크 상의 `SQLite` 또는 `PostgreSQL` 정형 테이블에 영구 저장하는 데이터 적재(Batch Load) 파이프라인 구축.
    *   API 호출 전 로컬 DB에 해당 연도/기업의 데이터가 이미 수집되어 있는지 먼저 검사하고, 없을 때만 DART API를 호출하도록 설계(Read-Through Cache 구조).

### 4.5 재무 추이 시각화용 구조화 데이터 포맷(JSON) 제공
*   **필요성**: 텍스트 답변도 좋지만 사용자는 재무 수치를 차트나 표로 한눈에 보고 싶어 합니다.
*   **구현 방안**: `/ask` 응답 객체에 마크다운 텍스트뿐만 아니라 프론트엔드에서 Recharts나 Chart.js 등으로 쉽게 시각화할 수 있는 시계열 포맷(예: `[{"year": 2022, "revenue": 302...}, {"year": 2023, "revenue": 258...}]`)의 구조화 데이터를 함께 담아 리턴하도록 응답 스키마 고도화.

### 4.6 미국 기업 공시(SEC EDGAR) 및 다국어 확장
*   **필요성**: 국내 기업뿐만 아니라 미국 상장사(Apple, NVIDIA, Tesla 등)의 재무 분석 수요가 매우 높습니다.
*   **구현 방안**: SEC EDGAR API를 연동하는 미국 공시 클라이언트(`edgar_client.py`)를 개발하고, 국문/영문 표준 계정과목을 통합 매핑하여 글로벌 기업 비교 질문까지 커버하는 하이브리드 에이전트로 도약.

## 7. 아키텍처 및 구현 로드맵상의 위험 요인과 대응 방안

프로젝트를 Phase 1부터 Phase 6까지 안정적으로 구현해 나가는 과정에서 만날 수 있는 리스크 요인들입니다.

1.  **DART API 응답 속도 병목**:
    *   *위험*: DART 서버의 일시적 지연이나 100ms API 요청 대기 시간으로 인해 에이전트 실행 속도가 너무 느려짐.
    *   *대응*: 필요한 대형주 10개 기업의 최근 3개년 데이터(사실 수치 + 본문 텍스트)를 개발 초기 단계에서 로컬 개발 환경으로 일괄 수집(Ingest)하여 개발을 진행하고, DART 직접 호출은 폴백 옵션으로 제한합니다.
2.  **임베딩/리랭커 로컬 실행 리소스 부족**:
    *   *위험*: 개발 환경(로컬 PC)에 외장 GPU가 없는 경우, BGE-M3 로컬 임베딩 연산이 쿼리당 수 초 이상 걸려 정상적인 RAG 흐름 테스트가 불가능해짐.
    *   *대응*: 환경 변수 설정(`EMBEDDING_PROVIDER`)을 통해 로컬 연산 외에 `OpenAI text-embedding-3-small` 또는 `Cohere/Gemini API`를 교체해서 테스트할 수 있도록 추상화된 팩토리 패턴 코드 구조를 조기에 도입합니다.
3.  **검증 루프의 무한 루프(Infinite Retry Loop) 가능성**:
    *   *위험*: LLM의 지속적인 검증 실패로 인해 에이전트가 예산을 초과하여 무한 호출되며 비용 폭탄 및 Rate Limit 발생.
    *   *대응*: `state.py` 내의 루프 계약서(Loop Contract)에 `max_steps` 제한(예: 최대 5회) 및 토큰 버짓 한도를 엄격히 검사하고, 초과 시 즉각 중단하고 기 설정된 "우아한 실패(Graceful Degradation)" 템플릿(현재까지 확보된 부분 결과 제공 + 추가 질문 가이드 제공)으로 제어를 넘깁니다.

## 8. 결론

본 **filing-agent** 프로젝트는 단순히 AI 프롬프트를 조절하는 차원을 넘어, **데이터 엔지니어링(수집/캐싱/정규화) - AI 모델 서빙(LiteLLM/임베딩/리랭커) - 비즈니스 흐름 제어(LangGraph) - 신뢰성 검증(Harness) - MLOps(DeepEval/Langfuse/Docker)**로 이어지는 엔드투엔드 AI 제품 생명 주기를 모두 경험할 수 있는 뛰어난 설계를 갖추고 있습니다.

Phase 0의 동기식 구조를 비동기식 I/O 및 연결 풀(Connection Pool) 기반으로 전환하여 웹 서비스로서의 성능 완성도를 다지고, 계정과목명 시맨틱 매핑 및 대화 기억 보존 등의 고도화 과제를 차근차근 밟아 나간다면 실무 프로덕션 수준에 걸맞은 훌륭한 엔지니어링 포트폴리오가 완성될 것입니다.
