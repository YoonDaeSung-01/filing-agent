# 📘 filing-agent — DART 전자공시 기반 재무 질의응답 어시스턴트 · 프로젝트 계획서

> **한 줄 정의**: 한국 기업 공시(DART 전자공시)를 넣어두면, 질문에 따라 **공시 본문을 의미 검색**하거나 **재무 수치를 정형 조회·계산**해서, 출처와 함께 답하는 에이전트.
>
> **목적**: AI 엔지니어 직군 포트폴리오. 모델 학습(파인튜닝)은 의도적으로 제외하고, **RAG → 에이전트(도구 라우팅) → 하네스/루프 → 평가 → 운영**으로 이어지는 "AI 엔지니어가 실제로 하는 일"을 한 프로젝트로 증명한다.
>
> **기반 스택**: Python 3.12 · FastAPI · LangGraph · 벡터 DB · Langfuse · Docker · GitHub Actions · (데이터) OpenDART API
>
> **작성일**: 2026-06-24 (DART로 고정 · 회의적 검토 반영) · **상태**: 주제·데이터·도구 확정, 구현 대기

> ⚠️ **이 도구의 성격 (가장 중요한 프레이밍)**: 이건 **투자 조언 도구가 아니다.** "이 주식을 사야 하나?"엔 답하지 않는다. **공시에 적힌 사실을 정확히 찾아 추출·계산하는** 도구다("이 회사가 공시한 2024년 매출액은?" → 출처와 함께 답). README·면접·시스템 프롬프트 전반에서 이 선을 분명히 한다.

## 0. 이 계획서를 읽는 법

이 문서는 "처음부터 끝까지 따라 하면 완성되는 사용설명서"를 목표로 한다. 각 Phase는 아래 7개 칸으로 똑같이 구성된다.

| 칸 | 의미 |
|---|---|
| 🎯 목표 | 이 단계가 끝나면 무엇이 동작하는가 |
| ❓ 왜 하는가 | 이 단계가 채용/실무에서 왜 중요한가 |
| 🔧 할 일 | 실제로 수행하는 작업을 순서대로 |
| 🧠 핵심 개념 | 이 단계에서 반드시 이해하고 넘어가야 할 용어 |
| ✅ 완료 기준 (DoD) | "끝났다"고 말할 수 있는 객관적 조건 |
| 🆘 막히면 | 자주 막히는 지점과 우회법 |
| 💬 어필 포인트 | 이 단계를 면접·이력서에서 어떻게 말할지 |

> **원칙**: 한 Phase를 완전히 끝내고(=DoD 충족 + git 커밋) 다음으로 넘어간다. 각 Phase 끝에서 동작하는 상태로 커밋하면, 중간에 멈춰도 항상 "돌아가는 결과물"이 남는다.

## 1. 프로젝트 개요와 목표

### 무엇을 만드는가

한국 기업 공시(DART) 데이터를 대상으로, 질문에 **사실 근거를 들어** 답하는 어시스턴트다. 단순 RAG 챗봇과 다른 점은 세 가지다.

1. **도구 라우팅 에이전트**: 답을 바로 뱉지 않고, 질문 성격에 따라 "공시 본문 의미 검색", "재무 수치 정형 조회", "계산" 중 무엇을 쓸지 스스로 판단하고, 필요하면 여러 단계를 거친다.
2. **하네스/루프**: 답변이 기준(특히 숫자 답변이 실제 공시 수치·출처를 인용하는지)을 통과하는지 검증하고, 통과 못 하면 스스로 재시도한다. 폭주를 막는 예산·종료 조건이 있다.
3. **평가(eval)**: "프롬프트·검색·라우팅을 바꿨더니 더 좋아졌는가?"를 숫자로 측정하는 파이프라인이 내장돼 있다. 공시 수치는 정답이 명확해서 좋은 평가셋을 만들기 쉽다.

### 두 가지 목표

- **학습 목표**: RAG, 에이전트(ReAct·도구 라우팅), 하네스 엔지니어링, 루프 엔지니어링, 가드레일, LLM 평가, 관측, 컨테이너화/CI를 직접 손으로 구현한다.
- **포트폴리오 목표**: 흔한 "RAG 챗봇"에서 멈추지 않고, **에이전트 운영·평가 계층 + 정형/비정형 데이터 라우팅**까지 갖춘 드문 결과물로 차별화한다. 한국 기업(DART) 데이터라 한국 취업 시장에 와닿는다.

### 무엇을 만들지 *않는가* (스코프 경계)

- ❌ **투자 조언** — 사실 추출 도구이지 추천 도구가 아니다(위 프레이밍).
- ❌ **모델 파인튜닝/사전학습** — 의도된 제외 항목(별도 경험으로 이미 보유). [확장 아이디어](#12-확장-아이디어-여력이-되면)에 곁가지로만.
- ❌ **화려한 프런트엔드** — UI는 최소한으로 두고 백엔드/AI 엔지니어링에 집중.
- ❌ **대규모 분산 인프라** — Docker + CI까지만.

## 2. 주제와 데이터 — 핵심 설계 포인트

이 프로젝트의 데이터는 **성격이 다른 두 층**으로 나뉘고, 이 차이가 에이전트를 가장 잘 보여준다. **이게 단순 RAG 챗봇과 갈리는 핵심**이다.

### ① 정형 재무 데이터 (DART 주요계정)

공시의 재무 수치(매출액·영업이익·당기순이익·자산·부채 등)는 K-IFRS 표준계정 기반의 구조화 데이터로 제공된다(DART 주요계정 API). 정확한 값이라 **타입이 있는 구조화 조회 + 결정론적 검증**에 적합하다. 핵심은 **수치를 텍스트 뭉치가 아니라 타입 있는 값(계정·연도·값·단위·출처)으로 다루는 것** — LLM이 글에서 숫자를 눈으로 골라 베끼다 틀리는 실패 모드 자체를 없앤다.
- 예: "2024년 매출액은?", "전년 대비 증감률은?", "A사 + B사 합은?"
- ⚠️ 스코프는 의도적으로 좁힌다(아래 3장): **재무 계정은 5개(매출액·영업이익·당기순이익·자산·부채)로 제한, 기업은 공시가 깔끔한 제조 대형주 10개**. 데이터 엔지니어링이 아니라 AI 에이전트를 보여주는 게 목적이라, 정규화 늪에 빠지지 않도록 범위를 잘라낸다.

### ② 비정형 텍스트 (공시 서술 부분)

사업보고서 안의 서술 섹션(사업의 내용, 위험 요인, 이사의 경영진단 등)은 자유 텍스트다. 이건 **RAG(임베딩·검색)** 코퍼스로 쓴다.
- 예: "주요 사업 위험은?", "회사가 설명한 성장 전략은?"

### 핵심: 에이전트가 도구를 *골라서, 필요하면 모은다*

흔히 "서술형은 의미 검색, 숫자형은 정형 조회"로 갈린다고 생각하기 쉽지만, 실제 질문은 그렇게 무 자르듯 나뉘지 않는다. 예를 들어 **"A사 2024년 실적 악화 원인은?"** 은 숫자(financial_lookup)와 경영진단 본문(doc_search)을 *둘 다* 봐야 한다. 그래서 목표는 "질문을 한 도구로 분류"가 아니라 **에이전트가 필요한 도구를 골라 결과를 종합**하는 것이다. 이런 하이브리드 질문은 결함이 아니라 오히려 더 인상적인 데모이고, 평가셋에도 의도적으로 넣는다(Phase 5).

도구는 셋으로 설계하되, **수치 도구는 타입 있는 값을 반환**해 LLM이 숫자를 손으로 옮기는 실패를 차단한다:

| 도구 | 역할 | 데이터 층 |
|---|---|---|
| `doc_search` | 공시 서술 텍스트 의미 검색 | ② 비정형 |
| `financial_lookup` | 재무 수치를 **구조화 값(계정·연도·값·단위·출처)**으로 반환 | ① 정형 |
| `compute_change` | 조회+증감/합계를 **도구 내부에서** 한 번에 계산(LLM이 숫자를 옮기지 않음) | ① 정형 |

> 별도 범용 "계산기"를 두지 않는 이유: 가장 위험한 연산(전년 대비 Δ%)을 `compute_change` 안에 넣으면, LLM은 회사·계정·연도라는 **식별자만** 넘기고 조회·계산은 도구가 한다. `financial_lookup`도 타입 값을 주므로, 여러 값을 합치는 잔여 케이스조차 "깨끗한 필드 하나를 복사"하는 저위험 작업이 된다. (합계 등 다른 연산이 필요하면 같은 패턴으로 도구를 추가)

검증 루프(Phase 4)는 LLM-judge가 아니라 **조회된 실제 값과 결정론적으로 대조**해 숫자 답변을 검증한다. 싸고 견고하며, 면접에서 강한 이야기다.

## 3. 데이터 소스와 수집 방법 (DART)

### 데이터 소스: OpenDART (한국 전자공시)

무료 인증키가 필요하다. Claude Code가 스크립트로 수집한다.
- **인증키(crtfc_key)**: `opendart.fss.or.kr`에서 무료 발급. 모든 요청에 쿼리 파라미터로 첨부한다.
- **호출 한도**: 키당 일일 호출 한도가 있으므로 응답을 로컬에 캐싱한다(같은 데이터는 다시 받지 않는다).
- **주요 엔드포인트** (모두 `https://opendart.fss.or.kr/api/` 하위):
  - **고유번호**: `corpCode.xml` (zip) — `corp_code`(8자리) ↔ 회사명·종목코드 매핑. 한 번 받아 로컬에서 사용. ⚠️ `corp_code`는 **종목코드와 다르다**(먼저 확보 필요).
  - **단일회사 주요계정**: `fnlttSinglAcnt.json?crtfc_key=&corp_code=&bsns_year=&reprt_code=` — 재무상태표·손익계산서의 주요 계정(매출액·영업이익·당기순이익 등). 2015년 이후 제공.
  - **다중회사 주요계정**: `fnlttMultiAcnt.json` — 여러 `corp_code`를 한 번에.
  - **전체 재무제표**: `fnlttSinglAcntAll.json` + `fs_div=CFS`(연결) / `OFS`(별도).
  - **공시검색(목록)**: `list.json` — 기간·보고서유형으로 공시 목록 조회. `corp_code` 없으면 검색기간 3개월 제한.
  - **원문 공시서류**: `document.xml` — 사업보고서 원문(서술 섹션 추출용).
- **보고서 코드(`reprt_code`)**: `11011` 사업보고서 · `11012` 반기보고서 · `11013` 1분기보고서 · `11014` 3분기보고서.

### Claude Code가 수집하는 흐름

1. `corpCode.xml`을 받아 `corp_code` ↔ 회사명 매핑을 확보(한 번).
2. 대상 기업의 `fnlttSinglAcnt`(또는 `fnlttSinglAcntAll`) JSON을 받아 `data/raw/facts/`에 캐싱. **연결(CFS) 우선, 없으면 별도(OFS)로 폴백.**
3. 일부 기업의 최신 사업보고서 원문(`document.xml`)에서 **서술 섹션**을 받아 `data/raw/filings/`에 텍스트로 저장.
4. 전 과정에서 인증키를 첨부하고, 일일 한도를 고려해 캐싱한다.

### 가볍게 유지하는 스코프 (CPU 환경 고려)

전 종목·전 계정을 받으면 정규화 늪에 빠진다. **범위를 공격적으로 잘라낸다(이게 1순위 의사결정).**
- **기업: 제조 대형주 10개로 하드 캡.** 공시가 비교적 깔끔한 업종을 골라 정규화 예외를 최소화한다. (금융업은 주요계정 API 제외 대상이므로 피한다)
- **계정: 5개로 하드코딩 수준 제한** — 매출액·영업이익·당기순이익·자산·부채. `fnlttSinglAcnt`(주요계정) API만 쓰고 `fnlttSinglAcntAll`(전체)은 쓰지 않는다(전체는 늪이 깊다).
- **텍스트 코퍼스**: 무거운 건 사업보고서 원문 파싱이므로, **위 10개 기업의 최신 사업보고서 서술 섹션만** 받는다.
- 이 한 번의 컷이 정규화 늪·일정 압박·"시간의 60%를 파싱에 쓰는" 비율 문제를 동시에 잡는다. **목적은 데이터 엔지니어링이 아니라 AI 에이전트를 보여주는 것**임을 잊지 않는다.
- 이러면 RunPod 없이 로컬 CPU로 충분하다. (GPU는 로컬 임베딩/리랭커를 빠르게 돌릴 때만 선택적으로 사용; API로 우회 가능)

## 4. 이 프로젝트로 어필되는 기술 스택 (채용 매핑)

| 구현 요소 | 대응 기술 키워드 | 채용/면접 맥락 |
|---|---|---|
| 공시 본문 인제스트 + 임베딩 + 벡터 검색 | RAG, 임베딩, 벡터 DB | "AI 면접 단골 1순위" |
| 청킹 전략 + 리랭킹 | RAG 고도화 | "RAG 성능이 안 나올 때 어떻게 개선하나" |
| 정형/비정형 도구 라우팅 (LangGraph) | 에이전틱 AI, ReAct, 도구 호출 | "2026 최대 화두", 챗봇 vs 에이전트의 차이 |
| DART 재무 수치 구조화 조회 + 도구 내 계산 | 정형·비정형 데이터 통합 | 단순 RAG와 차별화되는 핵심 |
| (선택) MCP로 도구 노출 | MCP | 도구·데이터 연동 트렌드 감각 |
| 검증 루프(숫자 출처 확인) + 루프 계약 | 하네스/루프 엔지니어링 | 모델이 아니라 "환경·피드백 루프" 설계 |
| 프롬프트 인젝션 방어 + 입출력 검증 | 가드레일, LLM 보안 | "2026 채용에서 보안 역량 중요" |
| 재무 골든셋 + LLM-as-judge + 회귀 테스트 | LLM 평가, LLMOps | "LLM 서비스 품질을 어떻게 측정·관리하나" |
| Langfuse 추적 | 관측가능성 | 하네스 5계층 중 모니터링 |
| Docker + GitHub Actions | 컨테이너화, CI/CD | 모델을 "제품·운영"으로 만드는 역량 |

## 5. 전체 아키텍처

### 요청 한 건이 흐르는 경로

```
[사용자 질문]
     │
     ▼
[1. 가드레일 — 입력 검증]  ← 프롬프트 인젝션/악성 입력 차단
     │
     ▼
[2. 에이전트 (LangGraph) — 도구 선택 후 종합]
     │   필요한 도구를 고르고, 하이브리드면 여러 개를 모음:
     │   ├─ 서술형 → doc_search (공시 본문 의미 검색: 벡터DB → 리랭킹)
     │   ├─ 숫자형 → financial_lookup (구조화 값: 계정·연도·값·단위·출처)
     │   ├─ 증감/합계 → compute_change (조회+계산을 도구 내부에서)
     │   └─ 예: "실적 악화 원인?" → financial_lookup + doc_search 둘 다 종합
     │   (필요 시 도구를 여러 번 조합: ReAct 루프 / max_steps·예산 제한)
     │       └─ max_steps 도달 시 → 우아한 실패(부분 결과 + "확정 못 함" 명시)
     │
     ▼
[3. 답변 초안 생성 (LLM)]
     │
     ▼
[4. 검증 루프 (Verifier)]  ← 숫자 답변이 실제 재무 값+출처를 인용? 형식 OK?
     │       │
     │   통과 못 함 → 에이전트로 되돌림 (루프 계약의 예산/한도 내)
     │       │
     ▼   통과
[5. 가드레일 — 출력 검증]  ← 형식/민감정보 최종 점검
     │
     ▼
[6. 최종 답변 + 출처(공시 인용)]
     
   (전 과정이 Langfuse에 트레이스로 기록됨)
```

### 구성 요소 한눈에

- **API 계층** (FastAPI): 질문을 받고 답을 돌려주는 입구.
- **에이전트 계층** (LangGraph): 어떤 도구를 쓸지 라우팅하고 ReAct로 반복하는 두뇌.
- **데이터 계층(두 층)**: ① DART 재무 수치(정형 조회 도구가 사용) ② 공시 서술 텍스트(벡터 검색이 사용).
- **하네스 계층**: 검증 루프 + 루프 계약 + 가드레일.
- **평가 계층**: 재무 골든셋 + LLM-as-judge.
- **관측 계층** (Langfuse): 모든 호출의 비용·지연·실패 지점 기록.

| 코드 에디터 | 개발 | VS Code 권장 |

### 발급받아야 할 키

- **LLM API 키**: OpenAI 또는 Anthropic(추상화 예정).
- **DART 인증키(crtfc_key)**: OpenDART에서 무료 발급 — **필수**.
- **(선택) 임베딩/리랭킹**: API 또는 로컬 모델.

> 💡 **비용 팁**: 임베딩(BGE-M3)·리랭킹(BGE-reranker-v2)은 로컬 오픈 모델로 무료 처리 가능. 이건 모델 *학습*이 아니라 *추론*만 쓰는 것이라 "ML 안 하기" 원칙과 충돌하지 않는다. LLM 호출(답변 생성)만 유료 API를 쓰고 나머지는 로컬로 두면 비용이 거의 안 든다. DART는 무료다. (BGE-M3는 bge-small보다 무거워 CPU 임베딩이 느리니, 부담되면 임베딩만 Cohere/OpenAI 호스팅으로 빼도 된다.)

### 환경 변수 (`.env.example`)

```bash
# LLM
LLM_PROVIDER=openai            # openai | anthropic
LLM_API_KEY=sk-...
LLM_MODEL=gpt-4o-mini

# 임베딩
EMBEDDING_PROVIDER=local       # local(bge-m3) | cohere | openai
EMBEDDING_MODEL=BAAI/bge-m3

# 벡터 DB
VECTOR_DB=pgvector             # pgvector | qdrant | chroma
PG_DSN=postgresql://localhost:5432/filing_agent   # pgvector 사용 시

# 데이터 소스 (DART)
DART_API_KEY=                  # OpenDART 인증키 (필수)
DART_REPORT_CODE=11011         # 11011 사업 | 11012 반기 | 11013 1분기 | 11014 3분기
DART_FS_DIV=CFS                # CFS 연결 | OFS 별도

# 관측 (Phase 6)
LANGFUSE_PUBLIC_KEY=
LANGFUSE_SECRET_KEY=
LANGFUSE_HOST=http://localhost:3000
```

> `.env`는 절대 git에 올리지 않는다. 저장소엔 `.env.example`만 둔다.

**💬 어필 포인트** — "초기부터 설정·비밀값 분리, 린터, 테스트 구조를 갖춰 시작."

### Phase 1 — 데이터 수집 + 기본 RAG (가장 중요한 토대)

**🎯 목표** — DART 데이터를 수집해 `data/raw/`에 쌓고, **공시 서술 본문**으로 질문하면 근거와 함께 답하는 최소 RAG가 동작한다. (`POST /ask` → 답변 + 출처)

**❓ 왜 하는가** — RAG는 "AI 면접 단골 1순위". 이 단계만 끝나도 "동작하는 RAG"라는 포트폴리오 뼈대가 생긴다. 동시에 데이터 수집 파이프라인을 구축한다.

**🔧 할 일**
1. **DART 클라이언트** (`ingest/dart_client.py`): 인증키 첨부 + 요청 간 지연 + 캐싱. `corpCode.xml`로 고유번호 매핑 확보.
2. **재무 수집** (`ingest/facts.py`): 대상 기업의 `fnlttSinglAcnt`(연결 CFS 우선, 없으면 별도 OFS) JSON을 받아 `data/raw/facts/`에 저장. (표준계정 정규화는 Phase 3에서 도구화하며 보강)
3. **서술 수집** (`ingest/filings.py`): 대상 10개 기업의 최신 사업보고서 원문(`document.xml`)에서 서술 섹션(사업의 내용·위험요인 등)을 받아 `data/raw/filings/`에 텍스트로 저장.
4. **청커/인덱서**: 서술 텍스트를 청킹(우선 고정 크기+overlap) → BGE-M3 임베딩 → pgvector 저장(원문·출처를 메타데이터로).
5. **리트리버**: 질문 임베딩 → 유사 청크 top-k 검색.
6. **LLM 클라이언트**: LiteLLM 게이트웨이로 공급자 추상화. "제공된 근거 안에서만 답하고 출처 표기" 시스템 프롬프트.
7. `POST /ask`로 노출. 공시 본문에 있는 사실을 직접 물어 확인.

**🧠 핵심 개념** — 임베딩, 벡터 유사도 검색, 청킹, "오픈북 시험"으로서의 RAG, API 수집 시 인증키·일일 한도·연결/별도 구분.

**✅ 완료 기준** — 공시 본문에 있는 서술형 사실을 근거와 함께 맞게 답한다. 없는 걸 물으면 "자료에 없다"고 답한다(환각 억제).

**🆘 막히면** — 답이 엉뚱하면 (a) 검색이 관련 청크를 못 가져오는지 (b) LLM이 근거를 무시하는지 분리해 확인. 검색된 청크를 같이 출력해 디버깅. DART 원문이 복잡하면 우선 1~2개 섹션만 추출해 시작.

**💬 어필 포인트** — "DART API 수집 파이프라인 + 임베딩·벡터DB·검색·프롬프트를 직접 연결해 RAG를 구현, 환각 억제를 위해 근거 강제·출처 표기를 적용."

---

### Phase 2 — RAG 고도화 (청킹 전략 + 리랭킹)

**🎯 목표** — 공시 본문 검색 품질을 끌어올린다. 청킹을 개선하고 리랭커로 재정렬한다.

**❓ 왜 하는가** — "2026 RAG 실무 핵심"이자 "RAG 성능이 안 나올 때 어떻게 개선하나"의 단골 답. 단순 RAG와 차별화되는 첫 지점.

**🔧 할 일**
1. **청킹 개선**: 고정 크기 → 문단/문장 경계 존중. 사업보고서는 섹션 구조가 뚜렷하니 섹션 단위 분할도 시도.
2. **리랭커**: top-20을 넉넉히 가져온 뒤 BGE-reranker-v2(cross-encoder)로 재정렬해 top-5만 LLM에 전달.
3. **하이브리드 검색(포함)**: 벡터 + 키워드(BM25 또는 BGE-M3 sparse) + 메타데이터 필터. 공시엔 회사명·숫자가 많아 사실상 필수.
4. Phase 5의 평가로 개선 전/후를 수치 비교할 수 있게 설정을 기록.

**🧠 핵심 개념** — 청크 크기/overlap의 영향, 리랭킹, 하이브리드 검색, "lost in the middle".

**✅ 완료 기준** — 같은 질문 세트에서 리랭킹 후 관련 청크가 더 자주 상위에. 설정값이 조절 가능.

**🆘 막히면** — 효과가 미미하면 청킹부터(보통 더 큰 영향).

**💬 어필 포인트** — "청킹·리랭킹을 적용하고 평가셋으로 정량 비교해 검색 정확도를 개선."

---

### Phase 3 — 에이전트화 + 도구 라우팅 (LangGraph)

**🎯 목표** — 답을 바로 만들지 않고, **질문에 맞는 도구(doc_search / financial_lookup / compute_change)를 에이전트가 고르고, 하이브리드 질문이면 여러 개를 모아 종합**한다. 필요하면 여러 단계를 거친다.

**❓ 왜 하는가** — "2026 최대 화두". 챗봇과 에이전트의 차이를 코드로 증명. **정형/비정형 라우팅은 단순 RAG와 가장 크게 갈리는 지점**으로, 면접에서 가장 강한 이야깃거리다.

**🔧 할 일**
1. **도구 정의** (`agent/tools.py`):
   - `doc_search`: Phase 1~2의 본문 검색을 도구로 감싼다.
   - `financial_lookup(회사, 계정, 연도)`: `data/raw/facts/`에서 **타입 있는 구조화 값**을 반환(예: `{계정:"매출액", 연도:2024, 값:300900000000, 단위:"KRW", 출처:"..."}`). LLM이 텍스트에서 숫자를 베끼지 않게 한다. 여기서 **표준계정 정규화**(다른 계정명/분류 흡수)와 **연결(CFS)/별도(OFS) 처리**를 구현 → `ingest/facts.py`/`tests/test_facts.py`와 연계.
   - `compute_change(회사, 계정, 연도A, 연도B)`: 조회+증감/합계를 **도구 내부에서** 계산해 반환. LLM은 식별자만 넘긴다(가장 위험한 Δ% 연산을 도구로 흡수).
2. **상태 정의** (`agent/state.py`): 대화·중간 결과·도구 호출 기록.
3. **그래프 정의** (`agent/graph.py`): LangGraph로 "추론 → 도구 → 관찰 → 다시 추론" 순환 + 종료 조건 + max_steps/예산.
4. `POST /ask`가 에이전트를 거치게 교체. 서술형/숫자형/증감형/**하이브리드형**("실적 악화 원인?" → 수치+본문 종합) 질문으로 테스트.

**🧠 핵심 개념** — ReAct 루프, 도구 호출(function calling), 그래프 상태 머신, 계정명 변이·연결/별도, 에이전트 실패 모드.

**✅ 완료 기준** — 서술형은 `doc_search`, 숫자형은 `financial_lookup`(+`compute_change`)로 적절히 도구를 고른다. **하이브리드 질문에선 수치·본문 도구를 둘 다 호출해 종합**한다. 도구 호출 과정이 로그로 보이고, `financial_lookup`은 텍스트가 아닌 구조화 값을 반환한다.

**🆘 막히면** — 도구 선택이 틀리면 도구 description을 더 명확히. 도구 설명이 곧 에이전트의 사용설명서다.

**💬 어필 포인트** — "LangGraph로 ReAct 라우팅을 설계해 정형(수치)·비정형(본문) 데이터를 통합했고, 표준계정 정규화·연결/별도 처리 같은 실무 데이터 문제를 다뤘다."

### Phase 4 — 하네스 & 루프 (차별화 1)

**🎯 목표** — 에이전트를 감싸는 안전장치. 답변(특히 숫자)이 기준을 통과하는지 검증해 자기 교정하고, 폭주를 막는 예산·종료 조건을 둔다. 악성 입력 가드레일을 추가한다.

**❓ 왜 하는가** — 하네스/루프 = "모델이 아니라 환경·피드백 루프를 설계하는 역량". 가드레일 = "2026 보안 역량". 대부분의 포트폴리오에 없는 부분.

**🔧 할 일**
1. **루프 계약** (`agent/state.py`):
   ```json
   {
     "goal": "공시 근거로 사실을 답한다",
     "scope": ["doc_search", "financial_lookup", "compute_change"],
     "verifier": "숫자 답변은 실제 재무 값+출처(보고서·연도)를 인용, 서술 답변은 출처 1개 이상",
     "max_steps": 6,
     "budget_tokens": 8000,
     "stop_condition": "검증 통과 또는 max_steps 도달",
     "escalation": "max_steps/검증 한도 도달 시 '우아한 실패' 템플릿으로 응답"
   }
   ```
2. **검증 루프** (`harness/verifier.py`): 숫자 답변이 조회된 실제 값·출처와 일치하는지 검사 → 실패 시 부족분을 피드백으로 돌려 재시도. `max_steps`/`budget` 초과 시 강제 종료.
3. **우아한 실패(Graceful Degradation) 템플릿**: 에이전트가 헤맬 때 그냥 뻗지 않게, max_steps 도달 시 응답을 미리 기획해 둔다. 예: "확정된 답을 찾지 못했습니다. 다만 다음까지는 확인했습니다: [부분 결과]. 질문을 [회사/연도/계정]으로 좁혀 주시면 다시 시도합니다." — 부분 결과 + 무엇이 부족했는지 + 다음 행동을 함께 제시.
4. **가드레일** (`harness/guardrails.py`):
   - 입력: 프롬프트 인젝션 패턴 차단, 길이 제한, **투자 조언 요청은 사실 추출로 선회**("매수 추천 대신 공시 사실 제공").
   - 출력: 출처 포함 검증, 민감/형식 점검.
5. 흐름을 `입력 가드레일 → 에이전트 → 검증 루프 → 출력 가드레일`로 연결.

**🧠 핵심 개념** — 루프 계약, 검증 루프/자기 교정, 프롬프트 인젝션 방어, 숫자 답변의 근거 검증.

**✅ 완료 기준** — 근거 없이 숫자를 지어내려 하면 검증 루프가 잡아 재시도/거절한다. "이전 지시 무시" 류가 차단된다. 어떤 질문도 정해진 step/예산 안에서 반드시 종료하며, **막혔을 때 뻗지 않고 우아한 실패 템플릿(부분 결과 + 다음 행동)으로 응답**한다.

**🆘 막히면** — verifier를 완벽히 만들려 말 것. "숫자 답변에 출처 인용 있는지" 단순 규칙으로 시작해 강화.

**💬 어필 포인트** — "루프 계약(종료·예산)과 검증 루프로 신뢰성·안전성을 설계, 특히 숫자 답변의 출처 검증과 프롬프트 인젝션 가드레일을 구현 — 하네스 엔지니어링을 직접 구현."

### Phase 5 — 평가(Eval) 파이프라인 (차별화 2)

**🎯 목표** — "바꿨더니 좋아졌는가?"를 숫자로 답한다. 재무 골든셋을 만들고, 검색 지표와 답변 품질(LLM-as-judge)을 측정하며, 회귀 테스트로 품질 저하를 잡는다.

**❓ 왜 하는가** — "LLM 서비스 품질을 어떻게 측정·관리하나"(2026 LLMOps 단골). **일반 개발자와 AI 엔지니어를 가르는 가장 큰 신호.** 공시 수치는 정답이 명확해 평가셋 만들기가 특히 쉽다.

**🔧 할 일**
1. **골든셋** (`eval/goldset.jsonl`): 공시 기반 질문-정답-출처를 작성. **양보다 질 — 다섯 유형(특히 하이브리드·엣지케이스)을 의도적으로 배치**:
   ```json
   {"id":"q001","type":"lookup","question":"A사가 공시한 2024년 매출액은?","expected_value":"...","expected_sources":["A_사업보고서_2024"]}
   {"id":"q002","type":"calc","question":"A사 2023→2024 매출 증감률은?","expected_value":"...","expected_sources":["A_사업보고서_2023","A_사업보고서_2024"]}
   {"id":"q003","type":"routing","question":"A사가 공시한 주요 사업 위험은?","expected_tool":"doc_search","expected_sources":["A_사업보고서_2024_위험요인"]}
   {"id":"q004","type":"combine","question":"A사+B사 2024년 매출 합은?","expected_value":"...","expected_sources":["A_사업보고서_2024","B_사업보고서_2024"]}
   {"id":"q005","type":"hybrid","question":"A사 2024년 실적 악화 원인은?","expected_tools":["financial_lookup","doc_search"],"expected_sources":["A_사업보고서_2024","A_사업보고서_2024_경영진단"]}
   {"id":"q006","type":"edge","question":"연결재무제표가 없는 C사의 2024년 매출은?","note":"별도(OFS) 폴백 검증","expected_value":"...","expected_sources":["C_사업보고서_2024_별도"]}
   ```
   - **N 규모 솔직 프레이밍**: 40개 안팎은 "통계적 유의성"이 아니라 **방향성**을 본다고 명시한다. 면접 방어 논리: "양을 줄인 대신 엣지케이스(연결/별도 혼재, 하이브리드)를 의도적으로 배치해 **질적 평가**에 집중했다."
   - **싸게 N 키우기**: `lookup` 유형은 정답이 자동 검증되므로(아래 결정론 대조) 이 슬라이스만 값싸게 늘려 숫자 정답률 모수를 키운다.
2. **검색/라우팅 지표** (`eval/metrics.py`): **견고한 지표를 전면에** — 숫자 정답 일치(결정론 대조, 자체), 라우팅/도구 선택 정확도, Hit@k·MRR. 검색 품질(context precision/recall)·faithfulness는 **RAGAS 또는 DeepEval**로 측정(둘 다 RAG 표준 지표 제공, DeepEval은 pytest 친화라 회귀 테스트와 잘 맞음). (작은 N에서도 fuzzy judge보다 덜 흔들린다)
3. **응답 품질**: LLM-as-judge는 서술형에만, 표본으로 보조 지표로 사용(전면에 두지 않음). RAGAS/DeepEval의 faithfulness가 이 역할을 상당 부분 대신한다.
4. **회귀 테스트** (`tests/test_eval_regression.py`): 골든셋 점수가 기준선 밑으로 떨어지면 실패하는 pytest → Phase 6 CI에서 자동 실행.
5. Phase 2~3 개선을 이 지표로 **개선 전/후 비교**해 README에 표로(방향성 개선으로 서술).

**🧠 핵심 개념** — 정답이 명확한/모호한 출력의 평가, LLM-as-judge, Hit@k·MRR, **routing accuracy**, 회귀 테스트.

**✅ 완료 기준** — 명령 하나로 점수표(숫자 정답률·라우팅 정확도·검색 지표·judge 점수)가 출력. 설정 변경 시 점수 변화를 비교 가능. 회귀 테스트가 CI에서 동작.

**🆘 막히면** — 골든셋은 10개로 시작해 늘린다. judge가 들쭉날쭉하면 채점 기준을 더 구체적으로.

**💬 어필 포인트** — "숫자 정답률·라우팅 정확도·Hit@k·LLM-as-judge로 평가 파이프라인을 구축, 회귀 테스트로 CI에서 품질 저하를 자동 감지. 모든 개선을 정량 검증." — **면접에서 가장 강한 한 문장.**

### Phase 6 — 관측 & 배포 (운영 마무리)

**🎯 목표** — 모든 호출을 추적·계측(Langfuse), 컨테이너화(Docker), CI에서 린트·테스트·평가 자동 실행(GitHub Actions).

**❓ 왜 하는가** — "모델을 만든다"에서 "제품으로 운영한다"로 넘어가는 마지막 다리. 관측·컨테이너·CI는 단골 우대 사항.

**🔧 할 일**
1. **관측**(Langfuse): 에이전트 각 단계(도구 호출·LLM 호출)를 트레이스로 기록. 비용·지연·실패 지점을 대시보드에서 확인. **기본은 Langfuse 클라우드 무료 티어**(셀프호스트는 Postgres 등 부가 서비스를 요구해 1대 로컬에선 무겁다 — 굳이 안 한다). 더 가볍게 가려면 초기엔 구조화 로그/단순 트레이스로 시작해 나중에 붙여도 된다.
2. **컨테이너화**: `Dockerfile`로 앱 이미지화, `docker-compose.yml`로 앱 + pgvector(Postgres)를 띄운다. (Langfuse는 클라우드를 쓰므로 compose에 포함하지 않아도 됨)
3. **(선택) 벡터DB 교체 실험**: pgvector 대신 Qdrant를 붙여 필터 검색 성능을 비교(금융 필터 검색에 강점) — "데이터 소스/스토어 추상화" 경험 어필.
4. **CI**(`.github/workflows/ci.yml`): 푸시마다 `ruff` → `pytest`(표준계정 정규화·연결/별도 단위 테스트 포함) → 평가 스모크.

**🧠 핵심 개념** — 관측가능성, 컨테이너화, CI/CD, LLMOps 전체 그림.

**✅ 완료 기준** — `docker compose up` 한 번으로 앱이 뜬다. 푸시하면 CI가 자동 실행. Langfuse(클라우드)에서 한 질문의 전체 트레이스를 열어볼 수 있다.

**🆘 막히면** — Docker가 처음이면 앱만 먼저 이미지화, Langfuse는 클라우드 무료 티어로 시작.

**💬 어필 포인트** — "Langfuse로 계측하고 Docker·GitHub Actions로 컨테이너화·CI를 구성, LLM 애플리케이션을 운영 가능한 형태로 마무리."

## 9. 기술 스택 상세 & 선택 이유

| 영역 | 선택 | 이유 | 대안 |
|---|---|---|---|
| 언어 | Python 3.12 | AI 엔지니어링 표준, 본인 주력 | — |
| 패키지 관리 | uv | 빠르고 현대적 | poetry, pip |
| API | FastAPI | 비동기·자동 문서화, 본인 스택 | Flask |
| 데이터 소스 | OpenDART (DART) | 한국 전자공시·무료 인증키, 한국 기업·취업 시장 적합 | (확장) SEC EDGAR |
| 재무 데이터 처리 | 표준계정 정규화 + 연결/별도 처리 | K-IFRS 계정명·연결(CFS)/별도(OFS) 처리가 실무 포인트 | dart-fss 등 래퍼 |
| 벡터 DB | **pgvector(Postgres)** → (대안)Qdrant | 2026 신규 RAG 기본값 · Postgres 운영 경험, 금융 필터 검색엔 Qdrant 강점 | Chroma(프로토타입), Weaviate |
| 검색 방식 | **하이브리드(벡터 + BM25/sparse) + 메타데이터 필터** | 공시의 회사명·숫자 정확 일치에 사실상 필수 | 순수 벡터(품질↓) |
| 임베딩 | **BGE-M3** (다국어 · dense+sparse+multi-vector 한 모델) | 한국어 공시에 필요한 다국어 + sparse로 하이브리드 내장, MIT | Cohere embed-v4 · OpenAI 3-large(호스팅) |
| 리랭킹 | **BGE-reranker-v2** (임베딩→리랭킹 2단계) | 2026 OSS RAG 기본 쌍(BGE-M3 + reranker-v2) | Cohere Rerank |
| 에이전트 | **LangGraph** (1.0) | 그래프 기반 라우팅·상태 설계 명확, 사실상 표준 | Pydantic AI · OpenAI Agents SDK · DSPy |
| LLM 게이트웨이 | **LiteLLM** | 멀티 프로바이더를 한 인터페이스로(공급자 교체 용이), 키워드 강함 | 직접 SDK 호출 |
| 평가 | **결정론 숫자 검증(자체) + RAGAS/DeepEval** | 숫자 정답률은 자체 대조, 검색·faithfulness는 RAGAS/DeepEval(pytest 친화) | LLM-as-judge(보조) |
| 관측 | **Langfuse(클라우드) · OTEL-native** | OSS 리더, 셀프호스트는 5+ 서비스라 클라우드 무료 티어, OTEL로 미래 호환 | Arize Phoenix(로컬·RAG 디버깅) |
| 컨테이너/CI | Docker + GitHub Actions | 표준 | — |
| 테스트/린트 | pytest + ruff | 표준 | — |

> 면접에서 중요한 건 "왜 골랐고 대안·트레이드오프가 무엇인지"를 말하는 것. 이 표가 그 답변의 준비물이다.
>
> **스택 의도(중요)**: 이 프로젝트의 목적 중 하나는 *지금 실무에서 많이 쓰이는 스택을 직접 경험*하는 것이다. 그래서 "가볍게/싸게"만 좇지 않고, 채용 키워드가 강하고 2026년 현재 표준에 가까운 것들(pgvector·하이브리드·BGE-M3·LiteLLM·RAGAS/DeepEval·OTEL)을 의식적으로 골랐다. 단, "지금 잘 쓰인다"는 빠르게 바뀌므로 구현 직전에 각 라이브러리의 최신 버전·대안을 한 번 더 확인할 것. 정말 가볍게 가야 하면 임베딩만 호스팅 API로 빼 로컬 부담을 0으로 만들 수 있다(트레이드오프).

## 10. 전체 일정 / 마일스톤

> 1인 기준 추정(주당 10~15시간). **솔직히 6주는 공격적인 일정이다** — DART 데이터 트러블슈팅과 오픈소스 버전 호환성 문제(LangGraph·Langfuse 등)에서 일주일이 날아갈 수 있다. 그래서 ① 항상 데모 가능한 상태를 먼저 만들고, ② 여유분을 두고, ③ 밀리면 덜어낸다.

| 주차 | Phase | 산출물(마일스톤) |
|---|---|---|
| 0주차 | 걷는 해골 | 1기업·1계정 종단 경로(조회→답변), 첫 커밋 |
| 1주차 | Phase 0~1 | DART 수집(10기업·5계정) + 동작하는 기본 RAG, README 골격 |

| 3주차 | Phase 3 | LangGraph 도구(본문/수치/증감), 표준계정 정규화·연결/별도 |
| 4주차 | Phase 4 | 검증 루프 + 루프 계약 + 가드레일 + 우아한 실패 |
| 5주차 | Phase 5 | 재무 골든셋 + 견고 지표 + 개선 전후(방향성) |

**중간 점검 규칙**: 매주 끝에 DoD 충족 시 그 상태로 커밋·푸시. 밀리면 확장(Phase 6 선택, [확장](#12-확장-아이디어-여력이-되면))부터 덜어낸다. **Phase 5(평가)는 반드시 포함** — 차별화의 핵심.

1. **한 줄 소개 + 데모 GIF/스크린샷**: 질문→답변(출처·수치 포함), 특히 **도구 라우팅이 드러나는** 장면.

4. **개선 전/후 평가 표**: "리랭킹 후 Hit@5 X→Y", "라우팅 정확도 Z%", "숫자 정답률" 등. — *가장 눈에 띄는 부분.*

### 예상 면접 질문 & 답변의 뼈대

- **"챗봇과 에이전트의 차이는?"** → 정형/비정형 도구 라우팅, 다단계 질문 처리 사례.
- **"RAG와 파인튜닝 중 왜 RAG를?"** → 최신·사실 정보 반영, 비용, 유지보수. 파인튜닝이 나은 경우와 대비.
- **"RAG 품질이 안 나오면?"** → 청킹·리랭킹·하이브리드, 그리고 **평가셋으로 검증**한 경험.
- **"정형 데이터(숫자)를 LLM이 어떻게 다루나?"** → 표준계정 정규화·연결/별도 처리, 정형 조회 도구 + 계산기 + 검증.
- **"에이전트가 폭주하면?"** → 루프 계약(종료·예산), 검증 루프, 가드레일.
- **"LLM 서비스 품질 관리?"** → 숫자 정답률·라우팅 정확도·Hit@k·LLM-as-judge + 회귀 테스트 CI.

## 12. 확장 아이디어 (여력이 되면)

메인을 끝낸 뒤 하나씩. 안 해도 프로젝트는 완성이다.

- **공시 유형 확대**: 사업보고서 외에 분기·반기보고서, 주요사항보고서 등으로 시계열·이벤트 질의 추가.
- **MCP 서버화**: 도구들을 MCP 규격으로 노출해 다른 에이전트가 꽂아 쓰게.
- **멀티에이전트**: 검색 담당·계산 담당·검증 담당으로 역할 분리(LangGraph 확장).

- **파인튜닝 곁가지**: 작은 모델을 특정 출력 형식에 LoRA로 튜닝해 "RAG로 안 되던 형식을 파인튜닝으로 해결"한 비교 사례(모델 학습이라 선택).

## 13. 흔한 함정과 주의사항

- **"한 방에 다 만들려는" 함정**: Phase 단위로 동작하는 상태로 커밋하며 전진.
- **평가를 마지막으로 미루는 함정**: Phase 5를 절대 생략하지 않는다. 일찍 작은 골든셋이라도 만들면 개선 판단이 빨라진다.
- **DART 인증키 / 일일 한도**: 모든 요청에 인증키(crtfc_key) 필수. 키당 일일 호출 한도가 있으니 응답을 로컬에 캐싱(같은 데이터 재요청 금지).
- **연결(CFS) vs 별도(OFS)**: 같은 회사도 두 재무제표가 있고 수치가 다르다. 일부 기업은 연결재무제표가 없어 별도로 폴백해야 한다. **어떤 기준으로 답하는지 명확히**(보통 연결). 이 처리를 다뤄봤다는 것 자체가 깊이의 증거.
- **표준계정 정규화**: 회사·업종마다 계정 분류·명칭이 달라 표준계정과목으로 정규화가 필요. **금융업은 주요계정 API 제외 대상**이라 대상 기업 선정 시 유의.
- **corp_code ≠ 종목코드**: 고유번호(8자리)를 `corpCode.xml`로 먼저 확보해야 한다.
- **데이터 무게**: 사업보고서 원문 파싱이 무겁다. 텍스트 코퍼스는 소수 기업으로 한정, 재무 수치(JSON)는 넉넉히.
- **비용 폭탄**: 저렴한 모델로 시작, judge는 표본에만, 임베딩/리랭킹은 로컬. (DART 자체는 무료)
- **비밀값 유출**: `.env`·인증키를 커밋 금지. 커밋 전 `git diff` 확인.
- **투자 조언 선 넘기**: 매수/매도 추천 요청은 사실 추출로 선회(가드레일).
- **도구 description 소홀**: 라우팅 오류의 대부분 원인. 설명을 명확히.
- **README 부실**: 코드가 좋아도 README가 부실하면 포트폴리오로 안 읽힌다.

### 선제적으로 적어둘 회의적 관점 (면접 방어용)

아래 리스크를 README의 '한계' 섹션에 **먼저** 적어두면, 면접관의 압박 질문이 오히려 문제 해결 능력의 증거가 된다. ("이미 알고 있었고, 이렇게 대응했다")

- **"AI가 아니라 데이터 엔지니어링 프로젝트 아닌가?"** → 정규화 늪을 인지해 **기업 10개·계정 5개·주요계정 API로 범위를 잘라냈다**고 답한다. 목적은 데이터 정제가 아니라 에이전트라서, 의도적으로 정형 데이터 비중을 눌렀다.
- **"라우팅이 틀리면 어떻게 되나?"** → 질문이 깔끔히 안 갈린다는 걸 전제로, **하이브리드는 여러 도구를 종합**하게 설계했고 **max_steps 도달 시 우아한 실패 템플릿**(부분 결과 + 다음 행동)으로 뻗지 않게 했다.
- **"LLM이 숫자를 잘못 넣지 않나?"** → 그래서 `financial_lookup`을 **타입 있는 구조화 값**으로 반환하고 가장 위험한 Δ% 연산을 `compute_change`에 넣어 LLM이 숫자를 옮기지 않게 했다. 숫자 답변은 **결정론적으로 대조**해 검증한다.
- **"평가셋이 40개뿐인데 유의미한가?"** → **통계적 유의성이 아니라 방향성**임을 명시한다. 양을 줄인 대신 **엣지케이스(연결/별도 혼재, 하이브리드)를 의도적으로 배치**해 질적 평가에 집중했고, 자동 검증되는 `lookup` 슬라이스로 숫자 정답률 모수는 값싸게 키웠다.
- **"인프라가 1대 로컬에서 무겁지 않나?"** → Langfuse는 **셀프호스트 대신 클라우드 무료 티어**(또는 구조화 로그)로 시작해 무게를 피했다.
- **"6주 일정이 현실적인가?"** → 공격적임을 인정하고, **걷는 해골(walking skeleton)로 항상 데모 가능 상태를 확보**하고 **버퍼 1~2주**를 두며 밀리면 확장부터 덜어낸다.
