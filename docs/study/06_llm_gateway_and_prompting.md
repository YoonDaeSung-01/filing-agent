> LLM을 "어떻게 부르고, 어떻게 시키고, 어떻게 도구를 쓰게 하나".

## 6-1. LiteLLM (LLM 게이트웨이)

### 한 줄 정의
> 100+ LLM 공급자(OpenAI·Anthropic·...)를 **하나의 OpenAI 호환 인터페이스**로 추상화하는 게이트웨이.

### 작동 원리 / 왜
- 공급자마다 SDK·파라미터가 다름 → LiteLLM이 **통일된 `completion()`/`embedding()`** 로 감쌈.
- 공급자 교체가 **모델명 문자열만 바꾸면** 됨(`gpt-4o-mini` → `claude-haiku-4-5-...`).
- 이 프로젝트 원칙(CLAUDE.md): "직접 SDK 호출로 짜지 말 것" → 게이트웨이 단일 진입점.

### 이 프로젝트에서 실제 사용
- `llm/client.py`가 **외부 LLM/임베딩 호출의 유일한 진입점**:
  - `ask()` — `litellm.completion`(RAG 답변 생성).
  - `embed()` — `litellm.embedding`(청크·질문 임베딩).
- 모델·키는 `config.py`(`llm_model`, `embedding_model`, `llm_api_key`)에서.

### 면접 Q&A
**Q. LLM 공급자 종속을 어떻게 줄였나요?**
> 모든 LLM·임베딩 호출을 LiteLLM 게이트웨이 한 모듈로 모았습니다. 공급자를 바꿔도 모델명만 바꾸면 되고, 호출 코드는 그대로입니다. 비용·지연·재시도 정책도 한 곳에서 관리할 수 있습니다.

### 🎴 암기 카드
- LiteLLM = 멀티 공급자 → OpenAI 호환 단일 API
- 교체 = 모델명만
- 단일 진입점(llm/client.py)

## 6-2. 프롬프트 엔지니어링 (이 프로젝트의 환각 억제)

### 핵심 개념
- **시스템 프롬프트** — 모델의 역할·규칙·금지사항을 고정.
- **컨텍스트 주입** — 검색한 근거를 사용자 메시지에 함께.
- **근거 강제 + 출처 표기** — 환각 억제의 핵심.

### 이 프로젝트 시스템 프롬프트 (llm/client.py)
> "DART 자료 기반 사실 추출 어시스턴트. ① 제공된 자료 내에서만 답 ② 출처(자료명·연도) 항상 표기 ③ 없으면 '확인할 수 없습니다' ④ 투자 조언(매수·매도) 금지."

→ 4개 규칙이 각각 막는 실패: ①③ 환각, ② 근거 추적성, ④ 스코프 이탈(투자 조언).

### 면접 Q&A
**Q. 환각을 프롬프트로 어떻게 줄였나요?**
> 시스템 프롬프트로 "제공된 자료 내에서만, 출처 표기, 없으면 없다고" 답하게 강제했습니다. 추가로 숫자는 아예 프롬프트에 의존하지 않고 도구가 구조화 값으로 반환하게 설계해, LLM이 숫자를 만들 여지를 없앴습니다.

**Q. 프롬프트만으로 충분한가요?**
> 아니요. 프롬프트는 1차 방어고, Phase 4에서 **검증 루프**(숫자 답변이 실제 값·출처를 인용하는지 결정론적으로 대조)와 가드레일을 더해 신뢰성을 보강합니다.

### 🎴 암기 카드
- 시스템 프롬프트 = 역할·규칙 고정
- 근거 강제 + 출처 표기 = 환각 억제
- 프롬프트는 1차 방어 → 검증 루프로 보강(Phase4)

## 6-3. Function/Tool Calling (Phase 3 핵심)

### 한 줄 정의
> LLM이 자연어 답 대신, **정해진 스키마로 "이 함수를 이 인자로 불러줘"** 라고 요청하게 하는 기능.

### 작동 원리 (메커니즘)
```
1. 개발자가 도구 스키마(이름·설명·인자 타입)를 LLM에 함께 전달
2. LLM이 질문을 보고 "tool_calls"(함수명+인자 JSON)를 출력
3. 시스템(코드)이 실제 함수를 실행
4. 결과를 다시 LLM에 전달(ToolMessage)
5. LLM이 결과를 종합해 최종 답변 (필요하면 2~4 반복)
```
- LLM은 **함수를 직접 실행하지 않는다**. "호출 요청"만 하고 실행은 코드가 한다(안전·통제).
- **도구 설명(description)이 곧 사용설명서** — 라우팅 정확도의 대부분이 여기서 갈림.

### 이 프로젝트 (Phase 3 설계)
- 도구 3종: `doc_search`(서술 검색), `financial_lookup`(수치 구조화 조회), `compute_change`(증감 계산).
- 위험한 계산(증감률)은 **도구 내부**에서 → LLM은 식별자(회사·계정·연도)만 넘김.
- 설계 상세: `docs/phase3_design.md`.

### 면접 Q&A
**Q. Function calling이 뭔가요?**
> LLM이 자연어 대신 "이 함수를 이 인자로 호출해줘"를 구조화해 출력하는 기능입니다. 실제 실행은 코드가 하고 결과를 다시 LLM에 줍니다. 이걸로 LLM이 검색·계산 같은 외부 능력을 안전하게 쓰게 됩니다.

**Q. 도구를 잘못 고르면요?**
> 라우팅 오류의 대부분은 도구 설명이 모호해서입니다. 그래서 description을 명확히 쓰고("수치는 financial_lookup, 서술은 doc_search"), 하이브리드 질문은 여러 도구를 종합하게 설계했습니다. max_steps로 폭주도 막습니다.

### 🎴 암기 카드
- LLM은 "호출 요청"만, 실행은 코드
- 흐름: 스키마전달→tool_calls→실행→결과반환→종합
- description = 라우팅 정확도의 핵심
- 위험 계산은 도구 내부(LLM은 식별자만)

## 6-4. 구조화 출력 (Structured Output)

### 왜 중요한가 (이 프로젝트 철학)
- LLM이 **글에서 숫자를 눈으로 골라 베끼면** 틀린다(실패 모드).
- → 재무 수치는 도구가 `{company, account, year, value:int, fs_div, source}` **타입 값**으로 반환.
- 검증도 LLM-judge가 아니라 **조회된 실제 값과 결정론적 대조**.

### 면접 Q&A
**Q. LLM이 숫자를 틀리는 문제를 어떻게 다뤘나요?**
> 숫자를 LLM이 옮기지 않게 했습니다. financial_lookup이 타입 있는 구조화 값으로 반환하고, 증감률 같은 위험 연산은 compute_change 도구 내부에서 계산합니다. 숫자 답변 검증도 실제 조회값과 결정론적으로 대조해, 모델 판단에 의존하지 않습니다.

### 🎴 암기 카드
- 구조화 값 = 타입 있는 {값·단위·출처}
- LLM 숫자 베끼기 실패 모드 차단
- 검증 = 실제 값과 결정론 대조(judge X)
