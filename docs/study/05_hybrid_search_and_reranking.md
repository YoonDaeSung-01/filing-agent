## 5-1. BM25 (희소/키워드 검색)

### 한 줄 정의
> 단어 일치 기반의 전통적 랭킹 함수. **단어 빈도(TF) × 역문서빈도(IDF)** 에 문서 길이 정규화를 더한 것.

### 작동 원리 (메커니즘)
- **TF (Term Frequency)** — 질의어가 문서에 많이 나올수록 점수↑ (단, 포화: 많이 나와도 무한정 안 늘게).
- **IDF (Inverse Document Frequency)** — 흔한 단어(예: "회사")는 변별력↓ → 가중치↓. 희귀한 단어(예: "삼성전자")는 가중치↑.
- **길이 정규화** — 긴 문서가 단어를 많이 포함하는 걸 보정.
- "dense"(임베딩)와 대비해 **"sparse"** 라 부름: 어휘 차원의 희소 벡터(대부분 0).

### 왜 RAG에 BM25를 더하나 (dense의 약점 보완)
- 임베딩(의미)은 **정확한 토큰 일치에 약함**: 회사명·종목코드·숫자·영문 약어.
- 예: "SK하이닉스 2024" 같은 질의는 의미보다 **정확 일치**가 중요 → BM25가 강함.
- plan.md: "공시엔 회사명·숫자가 많아 하이브리드가 사실상 필수."

### 이 프로젝트에서 실제 사용
- `retrieval/retriever.py`:
  - `_tokenize()` — 소문자화 + 한글/영숫자 런 토큰화(`[0-9a-z]+|[가-힣]+`). 형태소 분석은 안 씀(의존성·속도).
  - `_get_bm25_index()` — 전 코퍼스(15,666 청크)를 메모리에 1회 적재해 `BM25Okapi` 인덱스 캐싱.
  - `_bm25_search()` — 질의 토큰으로 점수 → 메타데이터 필터 → top-N.

### 면접 Q&A
**Q. 임베딩이 있는데 왜 BM25를?**
> 의미 검색은 회사명·숫자·코드 같은 정확 일치에 약합니다. 공시는 그런 토큰이 많아서, 키워드 강한 BM25를 결합해 서로의 약점을 보완했습니다.

**Q. BM25의 IDF가 하는 역할은?**
> 흔한 단어의 가중치를 낮추고 희귀하지만 변별력 있는 단어를 키웁니다. "회사"보다 "삼성전자"에 더 큰 점수를 줘서 핵심어 일치를 우대합니다.

### 함정
- 한국어는 공백 토큰화가 형태소를 못 나눔("매출액은" ≠ "매출액") → 형태소 분석기(kiwipiepy 등)면 더 정확. 현재는 휴리스틱(추후 개선 항목).

### 🎴 암기 카드
- BM25 = TF×IDF + 길이정규화 = 키워드(sparse) 랭킹
- dense 약점(정확일치) 보완
- 한국어 형태소는 추후 과제

## 5-2. 하이브리드 검색 + RRF

### 한 줄 정의
> 벡터(dense)와 BM25(sparse) **두 검색 결과를 융합**해 각자의 강점을 합치는 것. 융합은 **RRF**로.

### RRF (Reciprocal Rank Fusion) — 작동 원리
- 문제: 코사인 점수(0~1)와 BM25 점수(0~수십)는 **스케일이 달라** 직접 더할 수 없다.
- 해결: **점수 대신 순위(rank)만** 쓴다.
  ```
  score(doc) = Σ over each ranking list  1 / (k + rank)
  ```
  - `rank`는 0부터, `k`는 평활 상수(보통 60).
  - 여러 리스트에서 **모두 상위**인 문서가 합산 점수가 높아 앞으로 온다.
- 장점: **점수 정규화 불필요**, 구현 단순, 잘 동작(robust).

### 예시 (직관)
- 벡터 1등 + BM25 5등 문서 vs 벡터 3등 + BM25 3등 문서 → RRF가 둘을 공정히 합산.
- 두 검색이 **합의**하는 문서가 자연히 상위로.

### 이 프로젝트에서 실제 사용
- `retrieval/retriever.py`:
  - `_rrf_fuse(ranked_id_lists, k)` — 순수 함수(테스트됨, `tests/test_retriever.py`).
  - `search()` 파이프라인: 벡터 top-N + BM25 top-N → RRF 융합 → 후보 → 리랭킹.
  - `config.py`의 `hybrid_enabled`, `rrf_k=60`로 on/off·튜닝.

### 면접 Q&A
**Q. 두 검색 결과를 어떻게 합쳤나요?**
> RRF로 융합했습니다. 코사인과 BM25는 점수 스케일이 달라 직접 합산이 어려운데, RRF는 순위만 써서 `1/(k+rank)`를 합산하므로 정규화 없이 안정적으로 합칠 수 있습니다.

**Q. 가중합(weighted sum)은 왜 안 썼나요?**
> 가중합은 두 점수를 정규화하고 가중치를 튜닝해야 하는데, 스케일·분포가 달라 까다롭습니다. RRF는 순위 기반이라 그 문제가 없고 구현도 단순해서 택했습니다. 필요하면 가중 RRF로 확장할 수 있습니다.

### 함정 / 트레이드오프
- RRF는 순위만 봐서 **점수 크기 정보를 버림**(1등과 2등의 격차 무시). 대부분 괜찮지만, 점수 격차가 중요한 경우엔 손해일 수 있음.
- `k`가 작을수록 상위 순위 영향↑.

### 🎴 암기 카드
- 하이브리드 = dense + sparse 융합
- RRF = Σ 1/(k+rank), k≈60, **정규화 불필요**
- 합의하는 문서가 상위로
- 가중합 대비: 튜닝·정규화 불필요

## 5-3. 리랭킹 — bi-encoder vs cross-encoder

### 한 줄 정의
> 1차 검색이 넉넉히 가져온 top-N 후보를, 더 정밀한 모델로 **재정렬**해 top-K만 LLM에 넘기는 2단계 검색.

### 핵심 대비 (면접 단골 中 단골)
| | bi-encoder (임베딩) | cross-encoder (리랭커) |
|---|---|---|
| 입력 | 질문·문서 **따로** 인코딩 | (질문, 문서)를 **함께** 인코딩 |
| 출력 | 각자 벡터 → 코사인 | 관련도 점수 직접 |
| 상호작용 | 못 봄 | 토큰 단위로 봄 |
| 속도 | 빠름(미리 계산 가능) | 느림(쌍마다 추론) |
| 정밀도 | 낮음 | 높음 |
| 용도 | **1차 대량 후보** | **소수 후보 재정렬** |

### 왜 2단계인가 (메커니즘적 이유)
- bi-encoder는 문서 벡터를 미리 저장 → 수만 개도 빠르게 1차 필터(재현율 확보).
- 하지만 질문↔문서 상호작용을 못 봐 **상위 정밀도가 아쉬움**.
- cross-encoder는 (질문,문서)를 같이 넣어 정밀하지만 **느려서 전체엔 못 씀**.
- → **1차 bi-encoder로 20개 추리고, cross-encoder로 그 20개만 재정렬** = 속도+정밀 둘 다.

### 이 프로젝트에서 실제 사용
- `retrieval/reranker.py`:
  - 모델 `BAAI/bge-reranker-v2-m3`(다국어 cross-encoder), `sentence-transformers`의 `CrossEncoder`.
  - **lazy 로딩**(`_get_reranker` + `lru_cache`) → 모듈 import만으론 torch 불필요 → 모델 없이 테스트 통과.
  - `_attach_and_sort()` 순수 함수로 정렬 로직 분리(테스트됨).
- `retrieval/retriever.py`의 `search()` — RRF 후보 top-20 → 리랭킹 → top-5.
- `config.py`의 `rerank_enabled`, `retrieve_top_n=20`, `final_top_k=5`.

### ⚠️ 실제 겪은 함정 (면접 스토리감)
- 처음 `FlagEmbedding`의 `FlagReranker`를 썼더니 **`AttributeError: XLMRobertaTokenizer has no attribute prepare_for_model`**.
- 원인: FlagEmbedding 1.4.0이 **transformers 5.x와 비호환**(신버전에서 `prepare_for_model` 제거).
- 해결: 같은 모델을 **sentence-transformers의 CrossEncoder**로 로딩하게 교체 → 해결.
- 교훈: "OSS 버전 호환성"이 실무에서 시간 잡아먹는 지점. 추상화 계층(리랭커 모듈)을 둬서 구현 교체가 쉬웠다.

### 면접 Q&A
**Q. bi-encoder와 cross-encoder 차이는?**
> bi-encoder는 질문과 문서를 따로 인코딩해 벡터를 미리 저장할 수 있어 빠르지만 상호작용을 못 봐 정밀도가 낮습니다. cross-encoder는 둘을 함께 넣어 토큰 상호작용을 봐서 정밀하지만 느립니다. 그래서 1차는 bi-encoder로 후보를 넓게, 2차는 cross-encoder로 소수를 정밀 재정렬하는 2단계를 썼습니다.

**Q. 리랭킹이 실제로 효과 있었나요?**
> 정성적으로는 상위 청크가 더 관련성 높게 정렬됐습니다. 정량 검증은 Phase 5 평가셋으로 리랭킹 on/off의 Hit@k·MRR을 비교할 계획입니다. 모델 선택도 "다국어 OSS 표준이라 채택, 효과는 평가로 증명"이라는 입장입니다.

**Q. 모델 선택은 어떻게 했나요?**
> BGE-reranker-v2-m3가 한국어 포함 다국어 OSS 표준이라 채택했습니다. 다만 버전 호환성 이슈(FlagEmbedding↔transformers)가 있어 sentence-transformers CrossEncoder로 로딩 방식을 바꿨고, 리랭커를 모듈로 추상화해 교체가 쉬웠습니다.

### 🎴 암기 카드
- 리랭킹 = 1차 후보 top-N → 정밀 재정렬 → top-K
- bi-encoder: 따로 인코딩, 빠름, 1차
- cross-encoder: 함께 인코딩, 정밀, 소수 재정렬
- 2단계 = 속도(bi) + 정밀(cross)
- 함정: FlagEmbedding↔transformers5.x 비호환 → CrossEncoder
- 효과 증명은 Phase 5 평가

## 5-4. 전체 검색 파이프라인 한눈에 (이 프로젝트)
```
질문
 ├─(임베딩)→ pgvector 코사인 top-20 ─┐
 └─(토큰화)→ BM25 top-20 ───────────┴─(RRF 융합)→ 후보 top-20
                                              │
                              (cross-encoder 리랭킹)→ top-5 → LLM
```
- 메타데이터 필터(회사·연도)는 벡터·BM25 양쪽에 적용.
- 각 단계 on/off·파라미터는 `config.py`로 외부화 → Phase 5에서 A/B 비교 대비.
