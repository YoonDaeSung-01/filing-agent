## 4-1. pgvector

### 한 줄 정의
> PostgreSQL을 벡터 검색 DB로 만들어주는 **확장(extension)**. `vector` 타입 + 거리 연산자 + ANN 인덱스를 추가한다.

### 왜 (별도 벡터 DB 대신)
- 이미 쓰는 **관계형 DB 하나로** 메타데이터(회사·연도) + 벡터를 함께 다룸 → 운영 단순(서비스 1개).
- SQL `WHERE`로 **메타데이터 필터 + 벡터 검색**을 한 쿼리에.
- 2026 RAG 기본값 중 하나(Postgres 운영 경험 재사용).

### 작동 원리 (메커니즘)
1. **확장 활성화**: `CREATE EXTENSION vector;`
2. **컬럼 타입**: `embedding VECTOR(1536)` — 차원 고정.
3. **거리 연산자** (이게 핵심):
   | 연산자 | 의미 | 쓰임 |
   |---|---|---|
   | `<=>` | 코사인 거리 | 이 프로젝트(의미 검색) |
   | `<->` | L2(유클리드) 거리 | |
   | `<#>` | 음의 내적 | |
4. **검색 쿼리**:
   ```sql
   SELECT content, (1 - (embedding <=> %s)) AS score
   FROM filing_chunks
   WHERE corp_name = %s            -- 메타데이터 필터
   ORDER BY embedding <=> %s        -- 코사인 거리 오름차순(가까운 순)
   LIMIT %s;
   ```
   `1 - 코사인거리 = 코사인유사도` 라 score는 클수록 유사.

### 이 프로젝트에서 실제 사용
- `ingest/indexer.py` — `CREATE EXTENSION` + `filing_chunks` 테이블 + HNSW 인덱스 생성(`setup_table`), 배치 임베딩 적재(`index_chunks`).
- `retrieval/retriever.py` — `_vector_search()`가 위 쿼리로 top-N 코사인 검색.
- 파이썬 연동: `pgvector.psycopg`의 `register_vector(conn)`로 numpy 배열 ↔ vector 타입 자동 변환.

### 면접 Q&A
**Q. 왜 Pinecone 같은 전용 벡터 DB 대신 pgvector를?**
> 데이터 규모가 데모 수준(1.5만 청크)이고, 회사·연도 메타데이터 필터와 벡터 검색을 한 DB에서 처리하고 싶었습니다. Postgres 하나면 운영이 단순하고, 필터가 더 중요해지면 Qdrant로 갈아탈 수 있게 검색 계층을 추상화해뒀습니다.

**Q. 벡터를 어떻게 저장하나요?**
> `VECTOR(1536)` 컬럼에 넣고, HNSW 인덱스를 `vector_cosine_ops`로 만들어 코사인 검색을 가속합니다. 파이썬에선 numpy float32 배열을 register_vector로 바인딩합니다.

### 함정 / 트레이드오프
- `register_vector`는 **`vector` 확장이 생성된 뒤** 호출해야 함 → `setup_table`에서 확장 생성과 타입 등록 순서를 분리(이 프로젝트에서 실제 겪음).
- 차원 불일치(모델 바꿈)면 컬럼 재정의 + 재인덱싱.

## 4-2. ANN 인덱스 — HNSW vs IVFFlat

### 왜 인덱스가 필요한가
- 정확 검색(brute force)은 모든 벡터와 거리 계산 = **O(n)**. 수십만~수억이면 너무 느림.
- → **ANN(Approximate Nearest Neighbor)**: 약간의 정확도를 포기하고 **훨씬 빠르게** 근접 이웃을 찾는다.
- "재현율(recall) vs 속도" 트레이드오프를 인덱스 파라미터로 조절.

### HNSW (이 프로젝트 선택)
- **Hierarchical Navigable Small World** — 그래프 기반.
- 원리: 노드(벡터)를 **다층 그래프**로 연결. 위층은 듬성(고속 점프), 아래층은 촘촘(정밀). 위에서 시작해 가까운 이웃을 따라 점점 내려가며 탐색.
- 장점: **빌드 시 학습 데이터 불필요** → 동적 삽입에 유리, 높은 재현율.
- 단점: 메모리 사용 큼, 빌드 느림.
- 핵심 파라미터: `m`(노드당 연결 수), `ef_construction`(빌드 정밀도), `ef_search`(검색 정밀도).

### IVFFlat (대안)
- **Inverted File** — 벡터를 클러스터(centroid)로 나누고, 검색 시 가까운 몇 개 클러스터만 본다.
- 단점: 빌드 전에 **클러스터 학습(데이터 필요)** → 데이터가 어느 정도 쌓여야 함. 동적 삽입에 덜 유리.
- 장점: 메모리 적게 씀.

### 왜 HNSW를 골랐나
> 청크를 점진적으로 넣는 동적 인제스트라, 학습 데이터가 미리 필요 없는 HNSW가 맞았다. 재현율도 좋다. (메모리는 데모 규모라 문제 없음)

### 면접 Q&A
**Q. ANN이 뭐고 왜 쓰나요?**
> 정확 최근접은 O(n)이라 느려서, 약간의 정확도를 포기하고 빠르게 근접 이웃을 찾는 근사 알고리즘입니다. 재현율과 속도를 파라미터로 조절합니다.

**Q. HNSW와 IVFFlat 차이는?**
> HNSW는 다층 그래프 기반으로 학습이 필요 없어 동적 삽입에 유리하고 재현율이 높지만 메모리를 더 씁니다. IVFFlat은 클러스터 기반이라 빌드 전에 학습 데이터가 필요하고 메모리는 적게 씁니다. 저흰 점진적 인제스트라 HNSW를 택했습니다.

**Q. 재현율이 낮으면 어떻게 올리나요?**
> HNSW면 `ef_search`를 키워 검색 시 더 많은 후보를 탐색하게 합니다(속도와 트레이드오프). 인덱스 빌드 시엔 `m`, `ef_construction`을 키웁니다.

### 함정 / 트레이드오프
- HNSW는 정확 검색이 아니라 **근사** → 아주 드물게 진짜 최근접을 놓칠 수 있음. 그래서 1차는 넉넉히(top-20) 가져와 리랭킹으로 보정.
- 인덱스 파라미터는 "속도↔재현율" 손잡이. 정답은 도메인·규모에 따라 실측으로 정함.

### 🎴 암기 카드
- pgvector = Postgres + vector타입 + 거리연산자(`<=>`코사인) + ANN인덱스
- score = 1 − 코사인거리
- ANN = 근사 최근접(속도 위해 정확도 약간 포기)

- IVFFlat = 클러스터, 학습 필요, 메모리↓
