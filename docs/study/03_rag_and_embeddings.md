## 3-1. RAG (Retrieval-Augmented Generation)

### 한 줄 정의
> LLM에게 질문과 함께 **외부에서 검색한 근거 문서를 같이 넣어** 답하게 하는 패턴. "오픈북 시험".

### 왜 존재하나 (어떤 문제를 푸나)
LLM 단독의 3가지 한계를 푼다:
1. **지식 컷오프** — 학습 시점 이후·사내 데이터는 모른다. (이 프로젝트: 2024 공시)
2. **환각(hallucination)** — 모르면 그럴듯하게 지어낸다.
3. **출처 없음** — 왜 그렇게 답했는지 근거를 못 댄다.

→ 검색으로 **근거를 주입**하면 최신·사실 기반으로, **출처와 함께** 답할 수 있다.

### 작동 원리 (메커니즘)
```
[인제스트 = 미리 준비] 문서 → 청킹 → 임베딩 → 벡터DB 저장
                                                  │
[질의 = 실시간] 질문 → 임베딩 → 벡터DB 유사도 검색 → top-k 청크
                                                  │
                질문 + top-k 청크 → 프롬프트 → LLM → 근거 기반 답변 + 출처
```
핵심은 **"검색이 LLM의 작업기억(컨텍스트)에 무엇을 올리느냐"** 가 답 품질을 좌우한다는 것.
→ "Garbage in, garbage out": 검색이 관련 없는 청크를 주면 LLM도 틀린다.

### RAG vs 파인튜닝 (단골 비교)
| | RAG | 파인튜닝 |
|---|---|---|
| 새 지식 반영 | 문서만 추가 (즉시·저렴) | 재학습 필요 (비쌈) |
| 출처 제시 | 가능 | 어려움 |
| 사실 정확도 | 높음(근거 주입) | 환각 여전 가능 |
| 잘하는 것 | **지식 주입** | **형식/말투/스타일** |
- 결론: "최신·사실 지식"은 RAG, "출력 형식·톤 고정"은 파인튜닝. 둘은 **배타가 아니라 보완**.

### 이 프로젝트에서 실제 사용
- 인제스트: `scripts/ingest_all.py` → `ingest/chunker.py`(청킹) → `ingest/indexer.py`(임베딩+저장)
- 질의: `api/main.py`의 `POST /ask` → `retrieval/retriever.py`(검색) → `llm/client.py`(생성)
- 환각 억제: `llm/client.py`의 시스템 프롬프트 — "제공된 자료 내에서만 답하고 출처 표기, 없으면 '확인할 수 없습니다'".

### 면접 Q&A
**Q. RAG가 뭐고 왜 쓰나요?**
> 외부 문서를 검색해 LLM 컨텍스트에 근거로 넣고 답하게 하는 패턴입니다. 지식 컷오프·환각·출처부재를 해결하려고 썼고, 저희 프로젝트는 2024 공시처럼 모델이 모르는 최신 사실을 출처와 함께 답해야 해서 RAG가 맞았습니다.

**Q. RAG 답이 엉뚱하면 어디부터 보나요?**
> 두 가지를 분리합니다. (a) **검색이 관련 청크를 못 가져왔는지** — 검색된 청크를 같이 출력해 확인, (b) **LLM이 근거를 무시했는지** — 프롬프트에서 근거 강제가 됐는지 확인. 보통 (a)가 더 흔해서 청킹·리랭킹·하이브리드로 검색부터 고칩니다.

**Q. 환각은 어떻게 막았나요?**
> 시스템 프롬프트로 "제공된 자료 내에서만, 출처 표기, 없으면 없다고 답하라"를 강제하고, 숫자는 아예 LLM이 베끼지 않게 도구가 구조화 값으로 반환하도록 설계했습니다(Phase 3).

### 함정 / 트레이드오프
- 컨텍스트에 너무 많이 넣으면 **"lost in the middle"**(중간 내용 무시) + 비용↑ → top-k를 작게(리랭킹으로 정밀화).
- 검색 품질이 전부다. 좋은 LLM도 나쁜 검색은 못 구한다.

### 🎴 암기 카드
- RAG = 검색 + 생성 = 오픈북 시험
- 3대 동기: 컷오프 / 환각 / 출처
- 디버깅: 검색 실패 vs 근거 무시 **분리**
- RAG=지식, 파인튜닝=형식

## 3-2. 임베딩 (Embedding)

### 한 줄 정의
> 텍스트(또는 이미지 등)를 **의미를 담은 고정 길이 실수 벡터**로 변환한 것. 의미가 가까우면 벡터도 가깝다.

### 작동 원리 (메커니즘)
- 신경망(트랜스포머)이 텍스트를 받아 N차원 벡터를 출력. (이 프로젝트: **1536차원**)
- 학습으로 "비슷한 의미 → 가까운 벡터, 다른 의미 → 먼 벡터"가 되도록 공간을 만든다.
- **유사도 측정**: 두 벡터의 **코사인 유사도**(방향이 얼마나 같은가).
  - `cos(θ) = (A·B) / (|A||B|)` → 1에 가까울수록 유사. 크기(norm) 영향을 제거해 방향만 본다.
  - 거리로 쓸 땐 **코사인 거리 = 1 − 코사인 유사도**.
- 왜 유클리드 거리 대신 코사인? 텍스트 길이에 따라 벡터 크기가 달라질 수 있어, **방향(의미)** 만 보는 코사인이 보통 적합.

### bi-encoder (임베딩 모델의 구조)
- 질문과 문서를 **각각 따로** 인코딩 → 벡터 2개 → 코사인 비교.
- 장점: 문서 벡터를 **미리 계산해 저장** 가능 → 검색이 빠름.
- 단점: 질문↔문서 상호작용을 못 봐서 **정밀도가 낮음** → 그래서 리랭커(cross-encoder)가 필요(문서 05).

### 모델 선택 (이 프로젝트)
- 사용: OpenAI `text-embedding-3-small` (1536차원, 호스팅 API).
- 후보였던 것: **BGE-M3**(다국어·로컬·MIT, dense+sparse+multi-vector 한 모델).
- **왜 호스팅을 택했나** — 한국어 공시라 다국어가 필요한데, BGE-M3 로컬은 CPU에서 느림 → "임베딩만 API로 빼 로컬 부담 0" 트레이드오프(plan.md 명시).

### 이 프로젝트에서 실제 사용
- `llm/client.py`의 `embed()` — `litellm.embedding`으로 텍스트 리스트 → 벡터 리스트.
- `ingest/indexer.py` — 청크들을 배치로 임베딩해 pgvector에 저장.
- `retrieval/retriever.py` — 질문을 임베딩해 코사인 검색(`<=>` 연산자).

### 면접 Q&A
**Q. 임베딩이 뭔가요?**
> 텍스트를 의미가 담긴 고정 길이 벡터로 바꾼 겁니다. 의미가 비슷하면 벡터가 가까워서, 코사인 유사도로 관련 문서를 찾을 수 있습니다.

**Q. 왜 코사인 유사도를 쓰나요? 유클리드는요?**
> 텍스트 길이에 따라 벡터 크기가 달라질 수 있어, 크기를 빼고 **방향(의미)** 만 보는 코사인이 보통 더 안정적입니다. pgvector에선 `<=>`가 코사인 거리 연산자입니다.

**Q. 임베딩 차원이 크면 좋나요?**
> 표현력은 늘지만 저장·연산 비용과 "차원의 저주"가 따라옵니다. 3-small의 1536차원은 비용·품질 균형이 좋아 택했습니다.

### 함정 / 트레이드오프
- 임베딩 모델이 **바뀌면 차원·공간이 달라져** 전체 재인덱싱 필요 (`embedding_dim` 설정이 중요한 이유).
- 도메인 특수 용어(회사명·계정명)는 임베딩이 약함 → BM25 보완 필요(문서 05).

### 🎴 암기 카드
- 임베딩 = 의미 → 벡터, 가까움 = 유사
- 코사인 = 방향 비교(크기 무시), 거리 = 1−유사도
- bi-encoder = 따로 인코딩(빠름/덜 정밀)
- 모델 바꾸면 재인덱싱

## 3-3. 청킹 (Chunking)

### 한 줄 정의
> 긴 문서를 검색·임베딩 단위로 쪼개는 것.

### 왜 필요한가
- 임베딩 모델·LLM 컨텍스트엔 **길이 한계**가 있다.
- 너무 큰 청크 = 한 벡터에 여러 주제가 섞여 **검색 정밀도↓**.
- 너무 작은 청크 = **문맥 단절**(앞뒤 끊김).
→ 적절한 크기 + **overlap**(경계 문맥 보존)이 핵심.

### 작동 원리 + 이 프로젝트의 진화
- **Phase 1 (고정 크기)**: `chunk_size=500`마다 자르고 `overlap=100` 겹침. 단점: **문장 중간을 끊음**.
- **Phase 2 (경계 존중)** — `ingest/chunker.py`:
  1. 문단(빈 줄/줄바꿈) 경계로 먼저 분리
  2. 문단이 크면 **문장 경계**(`. ? !`, 한국어 `다.`)로 분리
  3. 그래도 크면 stride 기반 하드 분할
  4. 짧은 단위는 chunk_size까지 **그리디 패킹**, 청크 사이 overlap 꼬리 연결
- 불변식: 모든 청크 `len ≤ chunk_size`.

### 이 프로젝트에서 실제 사용
- `ingest/chunker.py`의 `chunk_text()` — 위 로직. `tests/test_chunker.py`로 경계 보존·overlap 검증.
- 재인덱싱: Phase 2 청킹으로 14,311 → **15,666개 청크**로 변화(경계 존중이 청크 구성을 바꿈).

### 면접 Q&A
**Q. 청킹 전략을 어떻게 정했나요?**
> 처음엔 고정 크기+overlap로 시작했는데 문장을 중간에서 끊는 문제가 있었습니다. 그래서 문단→문장 경계를 존중하고, 작은 단위는 chunk_size까지 묶되 청크 사이에 overlap을 주는 방식으로 개선했습니다. 효과는 Phase 5 평가셋으로 정량 비교할 계획입니다.

**Q. 청크 크기는 어떻게 정하나요?**
> 트레이드오프입니다. 크면 문맥은 살지만 한 벡터에 여러 주제가 섞여 검색이 둔해지고, 작으면 문맥이 끊깁니다. overlap으로 경계 문맥을 보존하면서 도메인(공시는 섹션이 김)에 맞춰 조정합니다.

### 함정 / 트레이드오프
- 한국어는 공백 토큰화가 형태소를 못 나눠 문장 경계 판단이 까다로움 → 정규식 휴리스틱으로 시작(`다.` 등), 형태소 분석은 추후.
- 청킹을 바꾸면 **재인덱싱(재임베딩)** 필요.

### 🎴 암기 카드
- 청킹 이유: 길이 한계 + 정밀도 vs 문맥 트레이드오프
- 고정크기 → 경계존중(문단→문장→하드)로 개선
- overlap = 경계 문맥 보존
- 청킹 변경 = 재인덱싱
