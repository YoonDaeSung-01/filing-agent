## 3. 데이터 소스 배치 (누가 무엇을 담당하나)

주가를 이미 FDR로 붙였고 이제 한투 키가 생겼으니, **역할을 명확히 나눈다**(중복·혼선 방지).

| 데이터 | 소스 | 이유 |
|---|---|---|
| **과거 일봉 차트·재무추이 스파크라인** | **FDR (유지)** | 키 없이 장기 OHLC. 이미 구현·캐시됨. 건드리지 않음 |
| **실시간/현재가·호가** | **한투 KIS (신규)** | 모의 주문 화면엔 실시간 현재가 필요. FDR는 지연·일봉 |
| **모의 주문·체결·잔고·수익률** | **한투 KIS vps (신규)** | 핵심. 실제 모의투자 엔진 |
| **공시·재무수치·AI Q&A** | **DART / filing-agent 코어 (유지)** | 이미 완성. 차별점 |
| **관심종목 뉴스** | **네이버 검색 API (신규)** | 한투엔 뉴스 없음 |

> 추상화 규칙(비전 [6-2] 계승): 한투·FDR·네이버를 각각 `MarketDataProvider`·`NewsProvider` 인터페이스 뒤에 둔다 → 나중에 소스 교체가 구현 1개 추가로 끝남.

## 4. 아키텍처 델타 (기존 재사용 + 신규)

```
src/filing_agent/                     (기존 코어 — 라이브러리처럼 재사용, 단방향)
  ├─ agent/ retrieval/ ingest/ llm/   ✅ 그대로 (공시 Q&A·재무·RAG)
  ├─ ingest/stock_client.py           ✅ FDR (오늘 추가분, 유지)
  └─ api/main.py                      ✅ 기존 엔드포인트 유지

src/filing_agent/platform/            (신규 플랫폼 계층 — 코어를 가져다 씀)
  ├─ auth/          JWT·비밀번호 해싱·본인확인 (단일 사용자 최소 구현)
  ├─ watchlist/     관심종목 CRUD
  ├─ market/        한투 클라이언트(토큰 캐싱·현재가·호가·주문·잔고·WS)
  ├─ paper/         모의투자(한투 vps 위임·포지션 집계·매매일지·주식모으기)
  ├─ news/          네이버 검색 수집·캐시
  └─ ai/            filing-agent 에이전트 호출 어댑터(종목→corp_code)

  ※ 경로 결정: `src/platform/`(top-level)은 ① 표준 라이브러리 `platform` 모듈과
     이름 충돌(shadowing), ② uv_build 가 `filing_agent` 단일 패키지만 빌드 →
     둘 다 피하려 `filing_agent.platform` 하위 패키지로 둔다. 모듈 경계·단방향 의존은 유지.

DB (기존 pgvector Postgres, 5433)     신규 운영 테이블 + Alembic 마이그레이션 도입
```

**신규 도입 필요**: `Alembic`(현재 마이그레이션 체계 없음), 한투 클라이언트, 인증 스택.
**신규 의존성**: `python-jose`(JWT), `passlib[bcrypt]`(해싱), `alembic`. (httpx·websockets는 이미 보유)

## 6. 데이터 모델 (신규 테이블 + Alembic)

비전 [9장] 계승. 기존 pgvector Postgres(5433)에 테이블만 추가하되, **수동 ALTER 금지 → Alembic 마이그레이션**으로.

```
users            (id, email, pw_hash, created_at)
watchlist        (id, user_id→users, ticker, corp_code?, memo, created_at)
paper_orders     (id, user_id, ticker, side, qty, price, status, kis_order_id?, created_at)
paper_positions  (id, user_id, ticker, qty, avg_price, updated_at)
trade_journal    (id, user_id, order_id→paper_orders, reason, review, created_at)
accumulation     (id, user_id, ticker, amount, cycle[weekly/monthly], next_run, active)  ← 주식 모으기
alerts           (id, user_id, ticker, type, threshold, active)                          ← Could
```

- `corp_code`는 AI 분석 대상(제조 대형주 10개)일 때만 채움 → **AI 기능은 corp_code 있는 종목에서만 활성화**(스코프 분리).
- 모의 체결의 정답 소스는 **한투 vps**. 로컬 `paper_*`는 표시·집계·일지용 미러 → 불일치 시 한투 잔고 기준 동기화.

## 7. 단계별 실행 (AP-0 ~ AP-7)

> Phase 라벨은 기존 filing-agent(0~6)·비전(A~F)과 구분해 **AP-n**으로. 각 단계 독립 데모 + `ruff`·`pytest` 통과 + 커밋.

| 단계 | 목표(끝나면 되는 것) | 핵심 작업 | 의존 |
|---|---|---|---|
| **AP-0 · 기반 정비** | 한투 붙일 토대 | ① `Alembic` 도입 ② `config`에 KIS/JWT/NAVER 필드 ③ `market/` 한투 클라이언트 스켈레톤 + **토큰 발급·캐싱**(만료·재발급) | — |
| **AP-1 · 한투 시세** | 실시간 현재가 표시 | 한투 현재가/호가 REST → `/stocks` 화면에 현재가 결합(과거 차트는 FDR 유지) | AP-0 |
| **AP-2 · AI 결합** ⭐ | 종목 화면에서 공시 Q&A·재무 | `ai/` 어댑터(종목명→corp_code), 종목 상세에 기존 에이전트 호출 + 면책 고지 | 코어(완료) |
| **AP-3 · 인증·관심종목** | 로그인 후 관심종목 저장 | `auth/`(JWT·bcrypt, 단일사용자 최소), `users`/`watchlist`, CRUD, 프론트 로그인 | AP-0 |
| **AP-4 · 모의투자** ⭐ | 가상 매수/매도·잔고·수익률 | 한투 vps 주문/잔고 위임, `paper_orders`/`positions`, 수익률 집계, 매매일지. **D2-1(해외 지원) 검증** | AP-0,3 |
| **AP-5 · 주식 모으기** | 정기·소수점 정액 매수 | `accumulation` 테이블 + `APScheduler` 정기 실행. 한투 소수점 지원 시 위임, 미지원 시 우리 체결 시뮬 | AP-4 |
| **AP-6 · 뉴스** | 관심종목 뉴스 피드 | 네이버 검색 `news/` + 캐시(종목별 TTL) | AP-3 |
| **AP-7 · 통합·배포** | 대시보드 + 배포 | 관심종목 통합 피드(주가+뉴스+공시), 포트폴리오·매매일지 시각화, docker-compose 확장, CI에 신규 테스트 | 전부 |
| **AP-8 · 교육·학습** ⭐ | 초보자용 정보·자료 + 맥락형 학습 | 학습 탭(용어집·기초·공시읽는법), **맥락형 툴팁**(숫자·용어 옆 ⓘ), 매매일지 AI 회고 | 대부분 정적 — **조기 착수 가능** |

**순서 근거**: 한투 토대(AP-0)를 깔면 시세(AP-1)·모의투자(AP-4)가 열린다. **차별점(AP-2 AI)은 인증 없이도 종목 화면에 바로 붙일 수 있어 조기 배치** → 초반부터 "흔한 모의투자 앱과 다름"을 보여준다. 무거운 모의투자(AP-4)는 토대·인증 뒤. AP-2와 AP-4는 서로 독립이라 순서 교체 가능. **AP-8 교육은 대부분 정적 콘텐츠라 의존성이 낮아 아무 때나(초반부터도) 끼울 수 있다.**

## 7-2. 교육·학습 자료 트랙 (AP-8 상세) — "초보자를 위한 학습 도구" ⭐

> 사용자 요청: **주식 처음 시작하는 사람을 위한 정보·공부 자료**. 이건 "투자 조언 아님" 정체성을 *강화*하는 트랙 — 우리는 추천이 아니라 **스스로 판단하도록 돕는다**.

### 제공할 자료 (카테고리)

| 자료 | 형태 | 경계 |
|---|---|---|
| **투자 용어집** (PER·PBR·ROE·배당수익률·시가총액·호가·거래량·지정가/시장가 등) | 정적 카드 + 검색 | ✅ 일반 지식 |
| **재무제표 읽는 법** (매출/영업이익/순이익 차이, 연결 vs 별도) — *우리 재무데이터로 실제 예시* | 정적 + 코어 연동 | ✅ 개념·사실 |
| **주식 기초** (거래시간, 주문유형, 세금·수수료, 결제일 T+2) | 정적 | ✅ 사실 |
| **공시 읽는 법** (사업보고서 구조, 위험요인 어디서 보나) — *filing-agent 정체성과 직결* | 정적 + AI 요약 | ✅ 사실 |
| **모의투자 시작 가이드**(온보딩) | 정적 | ✅ |
| **맥락형 툴팁** — 플랫폼 곳곳 숫자·용어 옆 ⓘ → 뜻 즉시 설명 | UI 레이어 | ✅ 그 자리에서 학습 |
| **매매일지 AI 회고** — "이번 모의매매에서 무엇을 배웠나" 정리 | LLM | ⚠️ 학습 정리까지만 |

### ⚠️ 경계선 (교육 콘텐츠 전용)

- ✅ "PER이 무엇인지", "영업이익과 순이익 차이", "지정가 주문이 뭔지" — **개념·사실**
- ❌ "PER 낮은 종목을 사라", "초보자 추천 종목 TOP5", "이 전략으로 사라" — **조언·추천**
- 매매일지 회고 AI: "무엇을 했고 결과가 어땠나" 정리는 ✅, "다음엔 이걸 사라"는 ❌

### 콘텐츠 출처 (저작권 안전)

- 용어·개념: **직접 작성(원본)** 또는 LLM 생성 후 검수. **타 사이트 복붙 금지**(저작권).
- 공신력 자료는 **링크로 연결**: 금융감독원 파인(fine.fss.or.kr), KRX, 금융투자협회 등 공개 교육자료(사실·출처 명시).

### UX 배치

- 새 탭 **`학습/가이드`** (용어집·기초·공시읽는법)
- + **맥락형 툴팁**: 각 화면 만들 때 함께 심음 → 플랫폼 전체가 학습 도구가 됨(별도 지루한 용어집 페이지에 그치지 않게)

## 8. 투자 조언 경계선 (신규 기능에 적용)

비전 [3장] 경계표를 그대로 강제. 신규 기능별 적용:

| 기능 | ✅ 하는 것 (사실·연습) | ❌ 안 하는 것 (조언·신호) |
|---|---|---|
| 주가·현재가·호가 | 수치 표시 | "지금 매수 타이밍" |
| 모의투자 | 체결·잔고·수익률·매매일지 | AI가 "이거 사라" |
| 주식 모으기 | **사용자가 규칙 설정**(종목·금액·주기) | 시스템이 종목 추천·자동 선정 |
| 뉴스 | "이런 기사가 있다"(사실 요약) | "호재/악재" 판정 → 매수신호 |
| AI 공시 Q&A | "공시에 ~라고 적힘" + 출처 | "그래서 좋다/사라" |
| 교육·학습 자료 | 용어·개념·사실 설명("PER이란") | 종목/전략 추천("PER 낮은 거 사라") |

> "주식 모으기"가 조언이 아닌 이유: **사용자가 조건을 설정**하는 자동화(가격 알림과 동일 성격). 시스템이 무엇을 살지 고르지 않는다. 기존 가드레일 노드("사야 해?" 차단)를 플랫폼 진입점에도 재사용.

## 10. 착수 즉시 할 일 (첫 커밋 단위)

**AP-0의 첫 조각**부터. 한 PR = 한 관심사:

1. `.env.example` + `config.py`에 **KIS/JWT/NAVER 필드 추가** (값 없음, 비밀값 자리만) — 안전·의존성 0
2. `uv add alembic python-jose "passlib[bcrypt]"` + Alembic 초기화(`alembic init`), 기존 pgvector DB 대상
3. `src/platform/market/kis_client.py` — 한투 **토큰 발급 + 디스크 캐싱**(만료 전 재사용, 실패 재시도) 최소 구현 + 단위 테스트(**한투 호출은 모킹** — 키·네트워크 없이 CI 통과)
4. `ruff`·`pytest` 통과 확인 → 커밋 `feat(platform): KIS 토큰 클라이언트 + 설정 스캐폴드`

> 이후 AP-1(시세)로 진행. 각 단계 시작 전, 만들 파일·역할을 요약해 확인받고 착수(CLAUDE.md 원칙).

## 11. 구현 지뢰밭 대비 (착수 전 못박기) ⭐

> 실전에서 실제로 터지는 4가지. 각 AP 단계 착수 시 반드시 확인·적용.

### 11.1 한투 vps 해외/소수점 미지원 대비 (D2-1 / AP-4)
- **경험칙**: vps 샌드박스는 국내주식 위주. 해외주문·소수점은 미지원이거나 호출 규격이 실전 API와 달라 작동하지 않을 가능성이 **매우 높다**.
- **AP-4 착수 당일 첫 작업**: KIS 공식문서 + **Postman으로 vps에서 (1) 해외 현재가 조회, (2) 해외 가상주문 API를 먼저 찔러본다.**
- 미지원 확인 시: 해외/소수점은 **로컬 시뮬(`paper_orders`/`paper_positions` + 자체 체결 엔진)로 즉시 피벗** — 주식모으기(AP-5) 로직과 통합. 국내는 한투 위임 유지. (억지로 vps에 맞추느라 시간 태우지 말 것.)

### 11.2 APScheduler 멀티 워커 중복 실행 (AP-5 / AP-7)
- `uvicorn --workers 4` 또는 compose 다중 인스턴스로 띄우면 스케줄러가 **프로세스 수만큼 중복 실행** → 주식이 원치 않게 2~4번 모아진다.
- **대응**: (a) 스케줄러를 **단일 프로세스에서만** 기동(전용 워커 또는 시작 플래그로 보장), 또는 (b) **`SQLAlchemyJobStore`로 DB 레벨 락**. 토이 규모면 (a)로 시작, 배포 확장 시 (b) 적용.

### 11.3 실시간 시세 중계 구조 (AP-1)
- ❌ **Next.js가 한투 WS에 직접 붙으면 `KIS_APP_KEY`가 브라우저에 노출** → 절대 금지.
- ✅ 구조: **한투 WS ⇄ FastAPI(수집·브로드캐스트) ⇄ Next(SSE 또는 WS)**. 키는 백엔드에만 둔다.
- 한투 WS **41종목 동시 등록 한도** → 사용자가 보고 있는 종목 세션을 백엔드에서 **레퍼런스 카운팅**으로 관리(아무도 안 보면 구독 해제, 초과 시 우선순위).

### 11.4 Alembic — 현재 코어는 sync psycopg (async 아님) → 일반 템플릿
- **확인 결과**: 코어는 `retrieval/retriever.py`·`ingest/indexer.py`에서 **raw psycopg(동기)** 사용. SQLAlchemy 비동기 엔진 **없음**.
- 따라서 신규 플랫폼 계층도 **sync SQLAlchemy로 통일**(코어 스타일 일치·단순). → **`alembic init`(일반 템플릿)** 사용, `-t async` **불필요**.
- FastAPI 동기 라우트는 threadpool에서 처리되므로 성능 문제 없음. (나중에 async 필요해지면 그때 전환 — YAGNI.)
